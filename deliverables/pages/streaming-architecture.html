Topic 03

流式架构

Palantir 的流式能力不是“把 Spark 改成 Streaming”，而是明确采用 Flink 和 Pipeline Builder 作为主要载体，并将流与批在上层目的地统一。

Conclusions

结论

流处理主引擎是 Flink，不是 Spark Structured Streaming

这意味着 Palantir 的实时能力依赖的是状态管理、Checkpoint 和 Watermark 体系，而不是把批处理 API 直接复用到流处理。

流与批的统一主要发生在“输出目的地层”

编程 API 和开发入口并未彻底统一，但输出都可以沉淀到 Dataset / Ontology，从而在上层应用侧形成无感消费。

Pipeline Builder 是流式能力的主入口

这说明流式产品化的关键不是提供很多 SDK，而是把状态、窗口、Join 和监控等复杂概念做成可配置能力。

Mechanisms

关键机制

1. 流式核心由状态与时间语义驱动

  • Keyed State 决定有状态算子的正确性与扩展方式。
  • Watermark 与 Allowed Lateness 决定窗口何时闭合、迟到数据如何处理。
  • Checkpoint 绑定恢复语义和 Offset 管理。

2. Stream 不是普通 Dataset 的实时开关

  • Stream 具有热缓冲与冷存储双层模型。
  • Pipeline Builder 和 Streaming Export 构成主要产品路径。
  • 流式支持 Java 与 Builder，但不支持 Python Transform。

3. 统一的关键在于结果层与应用层

Palantir 并没有强行让流和批在底层 API 上完全一致，而是通过 Dataset、Ontology 和上层应用保持结果层统一。这种“分层统一”比“API 强统一”更现实。

  • 秒级诉求走 Streaming Pipeline。
  • 分钟级诉求优先考虑 Incremental Batch。
  • 复杂历史计算仍然适合 Full Batch。

4. 流式监控和 Schema Evolution 是必要配套

一旦进入长期运行作业模式，延迟、堆积、状态增长、Schema 兼容性就不再是附属问题，而是产品可用性的核心部分。

5. 自建方案可以借鉴“热缓冲 + 冷存储”分层

调研中的 Kafka + Flink + Paimon 路线，说明这条产品思路是可复刻的，但产品化难点在于控制面与可观测性。

Implications

对我们的启发

  • 实时能力建设应基于明确的状态模型和时间语义，而不是把“支持流式”停留在引擎接入层面。
  • 无需强求流批 API 完全统一，更重要的是让上层对象和应用可以无感消费流批结果。
  • 实时链路的产品化必须同步建设监控、告警、Schema 演进与恢复机制，否则可用性会成为最大短板。
  • 分钟级诉求应优先用增量链路承接，避免所有实时需求都直接推向流处理。

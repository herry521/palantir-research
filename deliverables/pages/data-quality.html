Data Quality

Data Quality 质量控制面

本页把 Palantir Data Quality 调研压缩成可决策版本：它不是单点规则引擎，而是横跨构建期门禁、运行期健康、规模化监控、告警 issue 和治理生命周期的质量控制面。

Conclusions

核心结论

【推断】

Data Quality 是控制面，不是孤立产品

它由 Data Expectations、Data Health、Monitoring Views、通知、issue 和规则治理组成，共同覆盖构建前、构建中、运行后和协同处理。

【事实】

构建期门禁可以直接阻断坏数据

Data Expectations 绑定在 transform input/output 上随 build job 运行；FAIL 可 abort job，WARN 可继续并上报。

【事实】

运行期健康不等于 build abort

Health Checks 面向 dataset、schedule、table 的 status、time、size、content、schema 和 freshness 检查，核心职责是监控、展示、通知和 issue。

【推断】

强项在跨应用闭环

规则定义、CI 注册、构建执行、结构化结果、Health tab / lineage 展示、订阅告警、issue 和生命周期审查在同一平台模型内联动。

【建议】

自建要拆开结果模型和失败语义

BuildCheckResult、HealthCheckResult、MonitoringAlert、IssueLinkage 和 ExternalRoutePolicy 应分层建模，避免把门禁、监控和通知压成同一种规则。

Model

三层质量模型

Build-time expectationsTransform input/output 契约，FAIL 可阻断 build
Single-resource health checksDataset / schedule / table 的运行期健康
Scope-based monitoring rulesMonitoring Views 发现资源、路由告警和订阅
Notification / issue loopFoundry 通知、邮件、外部通道和自动 issue

构建期契约

Data Expectations 更接近 transform contract 的一部分，适合约束输出数据能否进入新 transaction，属于生产发布前的硬/软门禁。

运行期健康

Health Checks 负责已存在资源的健康、新鲜度、schema/content 漂移和 SLA 异常，适合发现生产态问题，但不应默认等同构建阻断。

规模化监控

Monitoring Views 用 static/dynamic scope 覆盖一批资源，让监控规则、severity、订阅和外部路由从单资源配置升级为资产域治理。

协同处理

Health Check failure 可以自动创建 Foundry Issue，resolve 后自动关闭；Monitoring View alert 可进入 Foundry、邮件、PagerDuty、Slack 或 webhook。

Closed Loop

质量闭环链路

规则进入工程生命周期 Code Repository branch -> PR review -> CI registers checks -> protected branch merge
构建结果进入平台上下文 Build job -> PASS / WARN / FAIL -> Builds / History / Data Health / Lineage
运行期问题进入处理闭环 Health check failed -> alert / notification -> Foundry Issue -> resolve / close
外部通知要单独控权 Viewer permission != external export policy; resource name may downgrade to RID

规则身份要稳定

不要把展示名当唯一主键。自建平台需要 rule_id / check_id、alias、tombstone、rename/delete 审计和历史保留。

结果要能解释失败

质量结果应携带 rule、expectation breakdown、build/job、branch、commit、dataset、transaction/view、trigger、threshold 和 duration。

排查要能跳上下文

失败页面必须能跳到 lineage、build history、job timeline、logs、rule diff 和最近数据/代码变更，否则质量规则只会变成告警噪声。

外部通道要可审计

Slack、PagerDuty、webhook、email 不能只记录 route；需要记录 export/redaction policy 和实际发送 payload 的脱敏结果。

Build Recommendation

自建平台最小模块

P0：Quality Rule Registry

注册规则、校验唯一性、绑定 dataset input/output、记录 owner、branch、commit、FAIL/WARN、retention 和生命周期审计。

P0：Build Gate Runner

在 build job 内执行规则，产生 PASS/WARN/FAIL/ABORT；它是防止坏数据进入新版本的最小生产门禁。

P0：Quality Result Store

拆分保存 BuildCheckResult 与 HealthCheckResult，并能按 dataset、branch、transaction、rule、owner 和时间追溯。

P1：Monitoring View Engine

用资源 scope 和 severity 组织批量监控、订阅、snooze、resolve、permission snapshot 和告警去重。

P1：Issue / Alert Router

连接质量结果、Foundry 内通知、自动 issue、PagerDuty、Slack、webhook 和邮件，并记录 dedupe 与 close/reopen 状态。

P1：External Export Controller

对外部通知中的资源名、敏感标签、组织边界、payload schema、secret reference 和 receiver audit 做独立管控。

Evidence Gaps

关键风险与证据缺口

【待验证】

内部结果 schema 未公开

公开资料没有披露 Data Expectations result 与 Health Checks result 是否共享内部 schema、ID 和保留策略。

【待验证】

非 Python 语言等价性不足

本轮主要确认 Python Transforms；Java、SQL、R 等路径是否具备完全等价的 Data Expectations 能力仍需补证。

【事实】

Health Checks build abort 未证实

当前只确认 Data Expectations 可 abort build；Health Checks 主要是运行期监控、告警和 issue 闭环。

【待验证】

外部通知细节仍需保守设计

PagerDuty、webhook、email payload、签名、重试、接收者权限、issue dedupe 和 reopen 机制仍未完全公开。

Topic 02

执行引擎与增量链路

Palantir 的执行层价值，不是选了 Spark 或 Faster 这样的某个引擎，而是把全量、增量、轻量和构建恢复组织成统一的运行模型。

Conclusions

结论

增量批处理是 Palantir 的关键中间层

它不追求行级 CDC，而是利用 Transaction 和文件级差量，把成本、时效和实现复杂度压在一个更可控的区间。

执行引擎的强项在于“托管与路由”

Compute Profile、独立 Build、失败回退和调度拓扑共同构成运行平台，而不是单次 Spark 任务封装。

Faster 说明执行层也可以做分层产品化

对中小规模、低延迟批任务，DataFusion 这类轻量引擎能作为 Spark 的补位，而不是替代。

Mechanisms

关键机制

1. Transaction 类型决定增量可行性

  • APPEND 适合增量处理，只读新增文件。
  • UPDATE 和 SNAPSHOT 往往触发全量重算。
  • semantic_version 和 require_incremental 让开发者显式控制重算边界。

2. Build 系统不是“调度器皮肤”

  • 平台基于依赖拓扑决定执行顺序与并行度。
  • 同一输出数据集存在写入串行和失败回滚保障。
  • Build History 与恢复语义共同构成运行可追溯性。
执行层最有价值的部分不是引擎本身，而是围绕版本、恢复、资源和调度构建出的平台能力。

3. Faster 作为 Spark 旁路，而不是全局替代

Faster 更适合运行时长短、中小规模且需要更低执行开销的任务；这是一种非常实用的“执行分层”思路。

4. 增量模型的边界必须被产品化暴露

何时增量、何时回退全量、何时强制失败，这些都需要被平台清楚表达，否则研发和运维无法形成稳定预期。

Implications

对我们的启发

  • 增量链路应当优先围绕 Dataset 版本与事务历史设计，而不是直接追求复杂行级 CDC。
  • 执行平台要把资源规格、失败回滚、重试、拓扑执行纳入一个统一 Build 模型，而不是散落到多个系统里。
  • 为中小规模任务设计轻量执行后端很有价值，但必须明确它与主执行引擎的边界和回退策略。
  • 增量语义需要被产品化暴露成开发者能理解的能力，不应作为底层黑盒存在。

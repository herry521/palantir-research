Topic 06

Dataset 权限与 Marking 架构

Foundry 的权限壁垒不是单个角色表，而是把资源角色、组织边界、Marking、血缘传播、细粒度策略、审批和审计放进同一个 Dataset 控制闭环。

Conclusions

结论

Dataset 权限不是单一 RBAC

完整判定由 resource role、organization、marking、classification、lineage-derived data requirements 和可选行级 / 属性级策略共同决定。

Role 和 Marking 必须分开

Role 决定能做什么，Marking 决定有没有资格接触某类敏感数据。Owner / Viewer 不能绕过 Marking。

血缘传播是核心分水岭

敏感要求会沿资源层级和直接数据依赖传播；只做 Dataset ACL 会漏掉派生数据的治理风险。

Permission Map

Dataset 权限全景

DAC

Project / Resource Roles

Owner、Editor、Viewer、Discoverer 等角色决定用户能否发现、打开、编辑或管理资源。

MAC

Organizations

组织边界应用于 Project / Space，用户通常满足至少一个相关 Organization 即可通过组织要求。

MAC

Markings

敏感数据资格要求，普通 Marking 按 all-of 理解，用户必须满足资源或数据上的所有 Marking。

CBAC

Classifications

高密级场景的 mandatory controls，支持层级密级和分类类别，通常需要平台级配置。

Policy

Restricted Views

在 Dataset 下游建立行级访问视图，适合“同一 Dataset 不同用户看到不同行”的场景。

Ontology

Object / Property Security

把 Dataset 要求延伸到对象和属性访问，让应用层也能执行数据安全边界。

Access Decision

访问判定模型

can_read_dataset_view = role_ok AND organization_ok AND file_markings_ok AND data_markings_ok AND classification_ok AND row_or_property_policy_ok

Resource access

判断用户能否看到、打开或管理这个 Dataset 资源，主要受 Project / folder / resource role、file marking 和 organization 控制。

Data access

判断用户能否读取 Dataset view 的实际内容，主要受上游 Dataset 通过 lineage 继承下来的 data requirement 控制。

可解释拒绝

权限系统要返回缺少 role、marking、organization 还是 classification，而不是只给一个通用 forbidden。

服务端强制

UI 可以展示 pill、badge 和提示，但真正的 allow / deny 必须在服务端 PDP 和查询执行层完成。

对象模型

MarkingCategory、Marking、MarkingMember、MarkingRoleAssignment、ResourceRequirement 和 DataRequirement 要分开建模。

权限分离

管理 Marking、应用 Marking、移除 Marking、成为 Marking member 是不同权限，不能默认互相包含。

两类传播

文件层级传播保护资源本身，数据血缘传播保护派生内容。后者是防止下游绕权的关键。

传播中断

stop_propagating / stop_requiring 应绑定 protected branch、required approver 和具体 Marking 权限。

控制面

  • 维护 Project、folder、Dataset、branch、transaction 和 lineage 元数据。
  • 维护 Marking category、member、apply/remove/manage 权限。
  • 提供 getAccessRequirements、whyDenied 和 simulateMarkingImpact。

执行面

  • Build 时读取 input requirements 并计算 output transaction requirements。
  • Query 时统一检查 role、resource requirement 和 data requirement。
  • 审批通过的 unmarking rule 只影响后续新 transaction。

Implementation Chain

端到端实现链路

01

创建 Marking

DPO / 平台管理员创建 category 和 marking，分配 members、apply/remove/manage 权限并写审计。

02

应用到 Dataset

校验资源 Owner 权限和 apply marking 权限，写入 ResourceRequirement，触发下游影响分析。

03

Pipeline 构建传播

BuildRun 解析 input transaction，合并 input requirements，写入 output transaction data requirements。

04

读取时强制判定

PDP 解析用户 entitlement，加载 resource 和 data requirements，拒绝时返回 missing requirements。

05

脱敏后移除继承

transform 声明 unmarking intent，protected branch 和安全审批人校验后，新 output transaction 才生效。

06

审计与模拟

记录成员变更、资源打标、读取、审批和移除动作，并支持 Marking 变更影响模拟。

Calculation Design

Marking 传递与计算设计

Apply-time

应用 Marking

校验 resource update-markings 权限和 marking apply 权限，写入 direct resource requirement，并触发 downstream impact simulation。

Resource hierarchy

资源层继承

effective_resource_requirements = direct + parent_effective，Project / folder 的要求物化到子资源。

Build-time

构建时传播

读取 input resource requirements 和 input transaction requirements，合并后写入 output transaction requirements。

Query-time

读取时判定

读取已物化的 resource + transaction requirements，和用户 Marking membership 做 all-of 判定。

carried_requirements(input) input.resource_effective_requirements ∪ input.transaction_requirements
output_transaction_requirements output.resource_effective_requirements ∪ union(carried - approved_unmarking_rules)
required_for_read dataset.resource_effective_requirements ∪ dataset_view.transaction_requirements

下游会继承要求，但不等于都被 direct marking

A 带 PII，B = clean(A)，C = aggregate(B) 时，B 和 C 的 transaction 会继承 PII requirement；但它们的 direct marking 可以为空，来源应记录为 lineage。多输入场景下，A 停止传播 PII 不代表 B 不再贡献 PII，所有携带该 Marking 的 input 都需要分别声明并审批。

Unmarking 必须 input-specific

多输入场景里，A 停止传播 PII 不代表 B 不再贡献 PII。传播中断规则应绑定 input -> output -> requirement。

读取路径不重新算 lineage

Lineage 传播应在 build-time 物化到 transaction requirements；query-time 只做快速 PDP 判定和可解释拒绝。

来源要保留

Requirement 去重时不能丢 source chain，否则 Access Debugger 无法解释某个 Marking 是哪个 parent 或 input 带来的。

事件流解耦

MarkingApplied、BuildStarted、UnmarkingRuleApproved、DatasetReadRequested 应分别触发重算、传播、审批和审计。

Build Your Own

自建平台建议

L0 / L1

Dataset ACL、Project role 和 group 授权只能解决协作访问，不能治理敏感数据传播。

L2

Marking 管理、Dataset direct marking 和 query-time enforcement 可以限制直接访问，但下游仍可能绕过。

L3 / L4

Lineage-aware propagation、transaction-level requirements、审批、Restricted View、SDS 和 Audit 才接近生产级闭环。

优先做

  • Resource / Dataset / Transaction / Lineage 元模型。
  • Marking 与 Marking 权限分离。
  • Build-time propagation 和 query-time enforcement。
  • Access debugger 和 audit log。

不要先做错

  • 不要把 Marking 做成普通标签。
  • 不要让 Owner 绕过敏感数据要求。
  • 不要让 unmarking 只靠代码声明生效。
  • 不要把历史 transaction 权限和最新权限混在一起。

Topic 04

血缘、Ontology 与治理

Palantir 最大的平台壁垒，往往不在算子或引擎，而在 Dataset 版本、Lineage、Marking、Ontology 和 Writeback 共同组成的治理闭环。

Lineage Ontology 治理

Conclusions

结论

数据版本和血缘是治理闭环的基础设施

没有 Transaction、Branch 和自动注册的依赖关系，血缘、影响分析、重放和审计都无法形成统一能力。

Ontology 把数据集提升成业务对象模型

这让平台不再只产出表，而是产出可被业务操作、可回写、可被应用消费的语义对象。

治理价值来自“自动传播”，不是手工登记

无论是血缘、标记、权限还是对象映射，都要尽量由平台自动捕获和传播，才能成为规模化能力。

Mechanisms

关键机制

1. Lineage 的抓手来自 Input / Output 声明

  • 编译期静态分析捕获 Dataset 级依赖。
  • 运行期记录实际读写版本，形成时间点血缘。
  • 列级血缘对手写代码依然存在天然边界。

2. Dataset Branch 支持实验，但不支持 Merge

  • 分支主要用于隔离试验和开发。
  • 数据分支不像代码分支那样可以直接合并。
  • 这迫使“回主线”动作必须通过 Transform 或发布流程完成。
Ontology 的价值在于把下游应用、对象模型和数据加工链路放到同一张图里理解。

3. Ontology + Writeback 构成业务闭环

Pipeline 输出 Dataset，经映射进入 Object Type；业务侧修改又可通过 Action / Writeback Dataset 回到数据链路，形成“分析结果可操作”的闭环。

4. 治理能力还包括 Marking、权限与安全策略传播

敏感标记、访问控制和审计如果能沿血缘自动传递，平台才能支撑更高敏感度的数据协作。

Implications

对我们的启发

  • 优先建设 Dataset 版本模型和自动血缘注册能力，这是后续治理与回放能力的基础。
  • 如果希望平台走向业务语义层，就不能只停留在 Dataset，需要逐步建立对象模型和回写入口。
  • 治理能力要尽量自动化传播，而不是依赖大量手工配置和人工维护。
  • Branch 的设计要区分“代码/配置分支”和“数据分支”，避免混淆 Git 语义与数据版本语义。

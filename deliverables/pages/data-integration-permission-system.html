Data Integration Security

Data Integration 权限控制面

本页把 Data Integration 权限体系调研压缩成可决策版本：核心不是再加一张数据权限表，而是把 source、credential、sync、transform、dataset/stream、API/export、approval 和 audit 放进同一套控制面。

Executive Summary

核心结论

【结论】

缺口是控制面，不是单点 ACL

Data Integration 权限要覆盖接入、凭据、同步、构建、消费、外发、审批和审计；只做 Dataset ACL 会漏掉 source preview、secret use、runtime egress 和 export。

【事实】

连接对象本身就是资源

Palantir 公开资料把 agents、sources、syncs、plugins、webhooks 和 external code connections 都建模为 resources，并允许叠加 roles、Organizations 和 Markings。

【推断】

P0 要先封住所有数据接触点

P0 不是高阶治理，而是统一对象模型、credential 一等对象、source capability 分权、commit-time 传播、query-time PDP/PEP、export/egress PEP 和审计快照。

【风险】

最危险的旁路在运行时和外发

高权限 service account、source editor 过宽、secret 暴露到代码、preview 自动等同 download/export、外部通知 payload 无 redaction，都会绕开 Dataset 层保护。

【共识】

四类专家已评审通过

Data Integration 架构、安全治理、平台工程、运维合规四个视角最终均为 Pass；未通过项已补齐后复审通过。

Roadmap

P0 / P1 / P2 建设路线

P0

先封底座

统一对象模型

source、dataset、stream、repo、job、credential、export 都有 RID、owner、requirements 和 audit id。

Credential 一等对象

secret.admin、secret.use、secret.read/expose_to_code 分开，默认 runtime-use-only。

Source capability 分权

preview、run_sql、sync_create、agent_assign、code_import、export_enable 不能被 source editor 隐式包含。

Build-time propagation

Sync/Transform commit 前写入 transaction_requirement，无法解释时 fail closed。

PDP / PEP 全覆盖

source preview、sync、build、query、download、API、stream、export、logs 都走统一决策。

Export / egress PEP

默认禁止任意 HTTP 出站；外发绑定 destination、credential、policy、redaction 和 audit。

Access decision snapshot

记录 entitlement snapshot、policy version、requirement version、credential scope 和 decision inputs。

P1

再补治理

Protected unmarking

stop_propagating / stop_requiring 绑定 input、output、branch、approval 和 approver 权限。

Access debugger

解释缺 role、marking、organization、classification、row policy 或 export policy。

External principal inventory

外部账号 owner、scope、external grants、last used、rotation 和 risk level 可查。

Time-bound access / SIEM

临时权限自动过期，高风险 export、break-glass、denied burst 可进 SIEM。

P2

最后做规模化

细粒度策略

Restricted View、row/property policy、field-level export redaction 叠加 Dataset requirements。

Recertification

按敏感等级周期复核权限，未确认自动降权或触发升级。

SDS 自动治理

scan -> apply marking / obfuscate / create issue / policy drift。

Policy-as-code

策略 diff、branch review、approval 和 rollback 纳入工程流程。

Runtime Boundary

运行时身份与外发边界

PR / Branch Build

检查触发者和 runtime principal

校验 repo/build 权限、input read、output branch write、Markings/Organizations，并写 actor、branch、input tx、requirements 快照。

CI / Publish

注册生产 job spec

CI principal 不能继承作者 token 任意写生产；必须校验 repo owns output spec、protected branch policy 和 ownership decision。

Schedule / Retry

锁定历史语义

scheduled build、retry、backfill 要记录 schedule owner、effective principal、config version、credential version 和 requirement version。

Stream

微批也要写 requirement

stream runtime principal 要校验 source/stream read、checkpoint owner、output write 和 requirement propagation。

Export / Webhook

外发不是 Viewer 扩展

export job 和 notification dispatch 必须绑定 route/destination policy、receiver ownership、secret scope、payload redaction 和 audit。

Logs

日志也是敏感面

build logs、webhook history、request/response、resource name 和 RID 要做独立 viewer 权限、retention 和 redaction。

Governance Loop

审批、审计与生命周期闭环

组合申请

Access request 不能只批 role；应能同时申请 role、Marking、Organization、export policy 或 source capability，并路由给对应 owner/admin。

可回放审计

P0 snapshot 至少包含 groups snapshot、Marking/Organization membership、credential scope、policy version、transaction requirement version 和 trace id。

外发审计

download/export/webhook/Kafka/SIEM 需要记录 export policy、destination credential、payload policy 和 redaction decision。

生命周期治理

TTL grant、break-glass、recertification、policy drift、external principal rotation 要能进入 SIEM 和 issue 流程。

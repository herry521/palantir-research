Pro-Code Capability

Palantir 高码能力研究

本页回答三个问题：高码能力到底强在哪里、怎么实现、怎么借鉴。所有关键结论按可信度标注为【事实 ref】、【推断】或【猜测】。

Strength

高码能力强在哪里

【事实 ref】

Code Repositories 是平台内 Git 工程环境

官方文档确认其支持 Git 分支、提交、tag、PR/code review、权限、IntelliSense、lint、error checking、preview、debug、impact analysis 和 repository upgrades。

【推断】

真正强点是代码被平台理解

Transform 的 Input / Output 把函数变成 Dataset 生产关系，使调度、血缘、质量、权限和版本都能围绕同一对象工作。

【事实 ref】

低码到高码有升级路径但不可逆

Pipeline Builder 可导出到 Java transforms repository，但部分转换不可导出，导出后不能自动回到 Builder，部分优化回退到 native Spark 后可能产生差异。

Implementation

怎么实现

Control Planeschedule / lineage / health / governance
【事实 ref】

Transform Contract 让代码进入 Dataset graph

Python、Java、SQL transform 都围绕输入输出声明组织生产关系；平台可据此生成血缘、影响分析和构建范围。

【事实 ref】

增量语义由 Dataset Transaction 承载

append-only 输入通常可增量；full rewrite、update/delete 文件、retention 边界和 snapshot 路径会影响增量可行性。

【推断】

调度中心更像 Dataset graph

Scheduling 支持按时间、数据更新、逻辑更新触发，并可围绕单个 Dataset、依赖或下游链路构建。这说明平台主轴是 Dataset 及其依赖图。

【猜测】

内部依赖索引服务边界未公开

公开文档能确认效果与入口，但跨 repository 依赖索引、事件传播延迟和内部 IR 细节仍需真实环境或更多资料验证。

Runtime

运行时、计算引擎与依赖

【事实 ref】

Python 路径最完整

Python transforms 支持 batch、incremental、共享库、data expectations，并可使用 Spark、DuckDB、pandas、Polars 等计算路径。

【事实 ref】

Java 与 SQL 有明确边界

Java 支持 batch、incremental、unit testing 和共享库；SQL 基于 Spark SQL，但官方文档明确 SQL transforms 不支持 incremental transforms。

【推断】

需要 Engine Router

自建平台应根据数据规模、增量需求、依赖库、资源规格和治理要求选择 Spark 或轻量引擎，避免把引擎选择变成用户自由拼装。

【推断】

依赖管理属于平台能力

共享库、artifact、运行时版本和 repository upgrades 应进入高码生命周期，否则会退化成每个项目自行维护的环境问题。

Governance

质量、血缘、权限与可观测性

【事实 ref】

质量检查靠近代码入口

unit tests、data expectations、lint、error checking、preview 和 debug 能在进入生产构建前缩短反馈环。

【事实 ref】

血缘与权限绑定 Dataset

Data Lineage、markings、roles 与 Dataset 生产关系关联，使代码变更能被映射到数据资产、下游影响和访问控制。

【推断】

治理不能只放在外部 CI

外部 CI 能验证代码，但缺少 Dataset graph、权限继承和下游消费者上下文，难以形成 Foundry 式闭环。

【事实 ref】

调度围绕数据新鲜度与链路范围

Scheduling 可按时间、数据更新、逻辑更新触发，并构建单个 Dataset、依赖、下游或两个 Dataset 之间的链路。

Interop

低码与高码互操作

【事实 ref】

Pipeline Builder 后端会写 transform code

公开文档显示 Builder 会生成后端 transform code，并在构建前执行 pipeline integrity checks。

【事实 ref】

导出是单向升级路径

导出到 Java transforms repository 后不可逆，部分转换不可导出，目标文件可能被覆盖，部分优化回退到 native Spark。

【事实 ref】

分支可以协作

Pipeline Builder branch 可与 Code Repositories branch 同名，使 Builder transform 从匹配 branch 读取输入 Dataset；缺失时可配置 fallback branches。

【推断】

共享 Contract 优先于共享代码

低码和高码应生产同一套平台 Contract；代码导出只是迁移手段，平台治理应消费 Contract / Dataset graph。

Borrow

怎么借鉴

P0：先做平台契约

定义 Transform Contract：输入、输出、参数、Schema、质量规则、执行引擎、增量语义和权限边界。【推断】

P1：再做 Dataset Transaction

每次写入生成不可变版本，运行时支持 added/current/previous 视图，让增量和回滚有平台依据。【推断】

P2：统一低码和高码底座

低码 Builder 应生成同一套 contract；代码导出作为升级路径，不承诺双向同步。【推断】

PoC

建议验证路线

【推断】

1. 最小 Transform SDK

提供 @transform(input=..., output=...) 风格声明，生成 JSON Contract，并支持 graph resolver。

【推断】

2. Dataset Transaction 原型

实现 append、replace、snapshot、atomic commit 和 version pointer，为增量和回滚提供数据层依据。

【推断】

3. Spark + lightweight 双运行时

建立 Engine Router 能力矩阵，明确哪些路径支持增量、质量规则、大规模分布式和快速 preview。

【推断】

4. Builder IR 与导出报告

低码 Builder 生成同一 Contract，并在导出时明确不可逆、不可导出算子、语义差异和所有权切换。

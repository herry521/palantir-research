Pro-Code Capability

Palantir 高码能力研究

本页回答三个问题：高码能力到底强在哪里、怎么实现、怎么借鉴。所有关键结论按可信度标注为【实时】、【推断】或【猜测】。

Strength

高码能力强在哪里

【实时】

Code Repositories 是平台内 Git 工程环境

官方文档确认其支持 Git 分支、提交、tag、PR/code review、权限、IntelliSense、lint 和 error checking，并支持 Python、Java、SQL transform repositories。

【推断】

真正强点是代码被平台理解

Transform 的 Input / Output 把函数变成 Dataset 生产关系，使调度、血缘、质量、权限和版本都能围绕同一对象工作。

【实时】

增量语义由 Dataset Transaction 承载

官方增量文档说明，append-only 输入通常可增量；full rewrite、update/delete 文件等会触发 snapshot 路径。

Implementation

怎么实现

Code RepositoryGit / PR / checks / preview
Transform ContractInput / Output / params / engine
Build RuntimeSpark / lightweight / Java / SQL
Dataset Transactionappend / replace / update / retention
Control Planeschedule / lineage / health / governance
【实时】

多语言不是平铺能力

Python 最完整，支持 batch、incremental、共享库、data expectations 和多种 compute engine；Java 支持 batch、incremental、unit testing；SQL 基于 Spark SQL，但不支持 incremental transforms。

【实时】

Pipeline Builder 可导出 Java code，但不可逆

官方明确导出会写入 Java transforms repository，部分转换无法导出，部分优化回退到 native Spark 后语义可能不同。

【推断】

调度中心更像 Dataset graph

Scheduling 支持按时间、数据更新、逻辑更新触发，并可围绕单个 Dataset、依赖或下游链路构建。这说明平台主轴是 Dataset 及其依赖图。

【猜测】

内部依赖索引服务边界未公开

公开文档能确认效果与入口，但跨 repository 依赖索引、事件传播延迟和内部 IR 细节仍需真实环境或更多资料验证。

Borrow

怎么借鉴

P0：先做平台契约

定义 Transform Contract：输入、输出、参数、Schema、质量规则、执行引擎、增量语义和权限边界。【推断】

P1：再做 Dataset Transaction

每次写入生成不可变版本，运行时支持 added/current/previous 视图，让增量和回滚有平台依据。【推断】

P2：统一低码和高码底座

低码 Builder 应生成同一套 contract；代码导出作为升级路径，不承诺双向同步。【推断】

Confidence

可信度规则

【实时】

本轮通过 Palantir 官方文档或当前仓库资料查证到的结论，等同于当前已验证事实。

【推断】

由多个已验证事实组合出的工程判断，逻辑链明确，但官方未直接以同一句话表达。

【猜测】

证据不足，只能作为后续验证假设，不应直接进入设计决策。

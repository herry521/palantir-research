Summary

先给结论

把“分支”和“fallback”拆成配置层、执行层、观察层三件事，避免把 runtime 解析结果误读成 OSet 观察结果。

Core Finding

fallback 是执行解析机制，不是观察层自动混合

JobSpec 和 input Dataset 可以按 fallback chain 命中其他 branch；但 output 只写 build branch，OSet / Lineage branch view 不会自动把 fallback main 数据展示成 feature 数据。

BuildRun records resolved branch Output stays on build branch OSet shows branch state
01

分支分三层

Global / Pipeline / Code branch 合并逻辑配置；Build branch 做运行时解析；Dataset branch 保存 transaction pointer。

02

fallback 粒度不是 Dataset 级开关

fallback list 更像 branch / workflow / build graph 级策略；per input dataset 只是运行时命中结果。

03

Sync 更像 raw producer

Data Connection 支持 sync 配置沙箱和 transaction 血缘，但不应等同于 Transform build 的任意分支写入能力。

Execution Layer

执行层形成混合版本链

开发者在 feature-x 修改 B 的 transform 和 Customer Object Type metadata，但没有构建 A(feature-x)。Build 会读取 A(main)，再写出 B(feature-x)。

InputA(main)fallback resolved
Buildfeature-xfeature logic
OutputB(feature-x)new transaction
OSetCustomer(feature-x)branch metadata

Build 证据应该记录

requested_branch=feature-x、resolved_branch=main、fallback_used=true、resolved_transaction=A_T10。

OSet 视图应该展示

Customer 的 feature metadata、B(feature-x) 的输出状态，以及 A(feature-x) 缺失；不要把 A(main) 伪装成 A(feature-x)。

用户最容易误解

看到 feature OSet 有数据，就以为整条链路都来自 feature branch；实际上输入可能来自 fallback main。

Build Your Own

自建平台要落哪些模型

1. fallback policy 不挂 Dataset

建成 branch_fallback_policy(scope, branch_name, ordered_fallbacks)，scope 可是 pipeline、code_repo 或 build_graph。

2. run 记录配置快照

BuildRun 保存 fallback_chain_snapshot，避免事后按当前配置反推历史运行。

3. input 记录解析结果

RunInput 必须有 requested_branch、resolved_branch、fallback_used 和 transaction/view。

4. output 和 OSet 分开

RunOutput 只写 build branch；OSet 记录 Object Type branch metadata、backing Dataset branch 和 indexing 状态。

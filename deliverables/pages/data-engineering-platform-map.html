Sync And Pipeline

数据同步与 Pipeline 专项关注

单独看数据同步和 Pipeline，重点不是多接几个源或多写几个算子，而是把 transform / expression 能力基线、状态、版本、事务、增量和质量变成平台内生能力。

Data Sync

数据同步需要补齐的能力

同步层解决“外部数据如何可信进入平台”，核心是连接器、状态管理、Schema 演进、CDC 语义和失败恢复。

Connector 插件模型 全量 / 增量 / CDC / 流式同步 Watermark / offset / cursor 托管 Schema Diff 与兼容性判断 幂等写入与失败重试 insert / update / delete CDC 语义 按时间、分区、offset 回放回填 坏行隔离与 dead letter 源端目标端 reconciliation Sync 配置版本化、分支与回滚 外部源到 Dataset 的血缘登记 连接密钥、敏感字段和同步审计
Pipeline

Pipeline 侧需要关注的能力

Pipeline 层解决“进入平台的数据如何被稳定加工”，核心是版本锁定、Transaction、增量判断、Build History 和发布门禁。

Input / Output 自动形成 DAG transform / expression 能力基线 算子注册中心与类型化 IR Build 读取输入 Dataset 固定版本 写出新的不可变 Transaction 识别 APPEND / UPDATE / SNAPSHOT 自动选择增量或全量重算 记录输入版本、输出版本和代码版本 按日期、分区、业务域参数化运行 历史区间回填与覆盖策略 质量检查不通过不发布版本 Transform 单测、样本测试、增量测试 dev / staging / prod 环境隔离 资源规格、队列、优先级和超时重试
Platform Boundary

底层存储和计算引擎需要定制么？

需要定制，但不建议优先魔改 Spark、Flink 或湖表内核。更现实的做法是在底层引擎之上建设 Dataset Control Plane， 统一封装事务、版本、Schema、血缘、质量、权限和消费位点。

存储层定制

  • 定义 APPEND / UPDATE / SNAPSHOT / DELETE 事务语义。
  • 通过 ReadSession / WriteSession 保证版本隔离和失败回退。
  • 在 Paimon / Iceberg / Delta 之上补 Dataset Branch、Schema Registry 和增量位点服务。
  • 治理小文件、compaction、snapshot rebuild，以及 Owner、SLA、权限、标记等平台元数据。

计算层定制

  • Spark 侧自动注入读写会话、输入版本锁定、输出事务提交和质量门禁。
  • Flink 侧托管 Checkpoint / Savepoint、offset、Exactly-once sink 和流式作业生命周期。
  • 调度器需要理解 Dataset 版本事件、依赖图、增量状态和回填范围。
  • DuckDB / DataFusion / Polars 可作为轻量引擎，但必须受 Dataset 事务模型约束。
Faster Engine

Faster / 轻量执行引擎规划

Faster 不应替代 Spark / Flink，而应作为 Pipeline 执行层的轻量后端：面向中小数据、低延迟、低成本和开发态快速预览。

P0

预留执行引擎抽象

Pipeline 节点不要写死 Spark，先抽象 Engine Router、Operator IR、ReadSession / WriteSession。

P1

支持轻量批处理

用 DataFusion / DuckDB / Polars 执行 Filter、Project、Join、Group By、Union 和简单表达式。

P1

支持 Spark fallback

某个节点不支持 Faster 时自动回退 Spark，保持同一 Pipeline 的完整性。

P2

支持 APPEND 增量

只处理新增文件 / 新增 snapshot，适合 append-only 数据，不承担复杂 upsert。

P2

成本与规模自动路由

小数据走 Faster，大 shuffle、大宽表、大状态计算走 Spark 或 Flink。

P3

查询优化与算子覆盖

增强列裁剪、谓词下推、Parquet row group pruning、表达式覆盖率和 UDF 边界。

适合 Faster 的场景

  • 中小数据量 Pipeline、执行时间小于 10-15 分钟。
  • 过滤、投影、聚合、轻量 Join、低成本高频小批次任务。
  • 开发态快速预览、append-only 增量处理。

不适合 Faster 的场景

  • 大规模 shuffle join、TB 级重计算、Streaming。
  • CDC upsert / delete、强状态计算、复杂 UDF。
  • 高级地理空间计算和超出单节点内存的数据处理。

Paimon 场景注意点

  • DataFusion 读 Paimon 优先支持 append-only / compacted snapshot。
  • DataFusion 写 Paimon 不建议作为早期目标，写入仍交给 Spark / Flink。
  • Faster 输出必须通过平台 WriteSession 提交 Dataset Transaction。

权限与治理

02

开发与执行层

开发表达统一

工程师、分析师、低代码用户都通过统一 Pipeline 模型、transform 节点和 expression 参数表达数据加工。

算子注册中心与类型化 IR

把 89 条 transform、335 条 expression、参数、类型、证据等级和实现状态纳入同一套控制面。

执行与调度体系

DAG、事件触发、失败重试、回填、资源规格和依赖执行统一产品化。

增量计算能力

基于数据版本判断 APPEND / UPDATE / SNAPSHOT，自动选择增量或全量重算。

流式能力

Stream Dataset、Flink Job、Checkpoint、流式质量监控和流式写业务对象。

01

资产底座层

Dataset 版本模型

统一 Dataset、版本、分支、Schema、Transaction、Owner、权限和血缘。

Media Set 资产模型

把 PDF、图片、音频、视频和 DICOM 作为可引用、可转换、可治理的非结构化资产纳入平台。

数据接入能力

稳定接入 JDBC、文件、API、CDC、Kafka / Pulsar，支持增量同步和 Schema 演进。

Focus Points

17 个能力建设关注点

这些关注点是评审平台方案时最应该反复追问的地方。它们横跨架构、产品、工程和治理，而不是某一个技术组件。

01

数据资产模型

Dataset 是否有统一的版本、分支、Schema、Transaction、Owner、权限和血缘。

02

数据接入能力

是否支持批、增量、CDC 和流式接入，并能处理 Schema 变更和接入监控。

03

Media Set 资产模型

非结构化文件是否有 media item、media reference、事务、版本、预览、转换、retention 和审计。

04

开发表达统一

避免 Spark、SQL、Flink、调度器各玩各的，让用户面对同一套 Pipeline 模型、transform 节点和 expression 参数。

05

算子注册中心

将 transform / expression 清单、参数、类型、运行环境、证据等级和实现状态集中管理。

06

执行与调度体系

支持 DAG、依赖调度、事件触发、失败重试、回填和资源规格选择。

07

增量计算能力

用数据版本和事务类型驱动增量执行，这是成本、时效和稳定性的关键。

08

流式能力

建设 Stream Dataset、Flink Job、Checkpoint、流式质量和低延迟对象写入。

09

数据质量与测试

同时覆盖产出后质量规则、开发态单测、样本测试、增量测试和 CI/CD。

10

血缘与影响分析

能追踪字段来源、Build 版本、下游影响和 Schema 变更风险。

11

权限与治理

支持 Dataset 级、字段级、行级权限，敏感标记传播、脱敏、审计和申请流程。

12

业务语义层

让应用、业务系统和 AI Agent 消费业务对象，而不是直接消费表。

13

用户回写闭环

业务用户修改进入 Writeback Dataset，再通过 Pipeline 合并回主链路。

14

可观测与健康度

统一查看 Build 状态、SLA、新鲜度、质量分、失败影响范围和资源成本。

15

环境与发布管理

dev / staging / prod 中，代码、配置、数据版本能一起晋级、回滚和对比。

16

资源与成本治理

Spark / Flink 资源需要队列、配额、优先级、成本归因和弹性策略。

17

AI / MCP 集成

后期接入自然语言生成 Pipeline、LLM 算子、Agent 操作和 MCP 暴露能力。

Build Order

三阶段落地顺序

建设顺序要防止两个极端：一边是只堆底层技术，另一边是过早复制 Palantir 的高级产品表象。

P0 最小闭环 01

第一阶段：先建底座

先把 Dataset 版本模型、数据接入、transform / expression 注册中心、Pipeline 表达、调度、基础质量和基础血缘建起来。

  • 统一 Dataset / Schema / Transaction 模型。
  • 预留 Media Reference 类型，避免文档、图片和音频后续绕行附件系统。
  • 导入 transform / expression final bundle，形成算子能力基线。
  • 打通接入、加工、调度、质量和血缘的最小链路。
  • 让现有 Spark / Flink / 调度体系通过平台边界复用。
P1 生产增强 02

第二阶段：补生产能力

围绕算子执行适配器、增量、流式、测试、监控、权限治理、资源管理，把平台从可用推到可规模化。

  • 将 transform 节点和 expression 参数接入类型校验、预览和运行时适配器。
  • 补齐 Media Set 的上传、转表、OCR、thumbnail 和 transactional output。
  • 将 APPEND / UPDATE / SNAPSHOT 纳入增量安全判断。
  • 补齐流式链路、Checkpoint、实时质量和失败恢复。
  • 建立 CI/CD、环境晋级、资源配额和成本归因。
P2 差异化闭环 03

第三阶段：做 Palantir 级差异化

在底座稳定后，把 Ontology、Writeback、业务对象、AIP / MCP 和 Agent 操作串成闭环。

  • 将数据资产提升为业务对象和对象关系。
  • 让业务对象稳定引用文档、影像、录音和视频，并通过 AIP / VLM 进行结构化抽取。
  • 让业务修改通过 Writeback 回到数据链路。
  • 让 AI 能在权限约束下读取、解释和触发平台动作。

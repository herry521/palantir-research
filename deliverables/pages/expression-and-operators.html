Topic 01

表达层与算子平台

表达层决定开发心智。Palantir 的第一层优势，不是把 Spark 封装得更漂亮，而是把“如何描述数据开发”做成了可复用的平台契约。

表达层 Transform DSL 算子平台
本页导航 结论 关键机制 对我们的启发 相关来源

Conclusions

结论

路径即依赖，是表达层最重要的抽象

通过 Input() / Output() 声明数据集路径，平台就能自动构建 DAG、感知依赖与调度关系，降低手工编排成本。

DSL 不只是语法糖，而是平台契约入口

装饰器、pipeline.py 注册入口和可发现的 Transform 集合，让开发逻辑天然可被平台理解、追踪和治理。

算子平台把“代码能力”转化成“产品能力”

当逻辑可以被沉淀为带契约的算子，低代码、代码优先和平台治理才能共享同一个能力底座。

Mechanisms

关键机制

1. Transform DSL 把开发动作平台化

  • @transform_df、@transform、@transform_pandas 对应不同开发范式。
  • 数据集路径而不是任务节点成为核心依赖抽象，更利于平台统一编排。
  • pipeline.py 负责注册和发现，使平台可以集中感知所有 Transform。

2. Pipeline Builder 与 Code Repository 并列存在

  • 代码优先路径服务复杂逻辑与复用。
  • 低代码路径服务标准化转换与可视化开发。
  • 二者共享数据集、调度和治理模型，避免裂成两个世界。
表达层的价值不在于前端交互，而在于它是否能向下连接执行层、向上连接治理与应用层。

3. 算子契约是走向平台能力的桥

从调研里的算子平台方案看，输入输出声明、参数声明、Schema 推导和执行绑定需要分层设计，才能同时支撑编译期校验和运行期执行。

4. 平台真正要沉淀的是可发现、可校验、可组合

Palantir 的做法说明，算子丰富只是结果，平台更底层的价值是让逻辑可被发现、被拼装、被治理和被复用。

Implications

对我们的启发

  • 优先统一“数据开发的表达模型”，比先做复杂可视化 Builder 更关键。
  • 把数据集路径、输入输出契约和 Schema 传播作为平台主线，而不是把调度节点当中心。
  • 低代码和代码优先应共享同一套底层算子与契约，否则后续治理和复用会持续分叉。
  • 算子平台设计要同时考虑编译期校验和运行期执行，避免只做“函数库”而缺失平台感知能力。

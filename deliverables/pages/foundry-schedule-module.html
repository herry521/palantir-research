Foundry Scheduling

Foundry Schedule 运行模式

本页把最新调研结论压缩成一个可决策版本：Foundry Schedule 的主语是 Dataset graph freshness，而不是 DataWorks 式业务周期实例。业务周期对齐调度与 freshness 调度必须拆开设计。

Conclusions

核心结论

【事实】

Schedule 的主语是 Dataset graph build

Trigger 满足后，平台动态解析 build 范围、branch、scope、staleness 和 build locking，再决定是否真正创建 build。

【事实】

Trigger 是状态机，不是业务时间 join

Time trigger 是瞬时墙钟条件；event trigger 是锁存 satisfied 状态；AND/OR 只组合 satisfied 状态。

【事实】

Run 成功不等于 Build 成功

Succeeded 只表示 build 被成功发起；Ignored 通常表示目标 up-to-date，没有创建 build。

Runtime Model

Schedule 到 Build 的运行链路

Trigger satisfiedtime / event / AND / OR 组合满足
Schedule run记录 run 状态，但不等价于 job 成败
Graph resolution动态解析 target、scope、branch、excluded
Staleness check输入版本或逻辑变化决定是否 stale
Dataset transactionbuild 写入输出 transaction 并形成版本血缘

它不是固定 job 清单

Schedule 保存的是触发条件和 graph selection rule。每次触发时，Foundry 会重新评估 included datasets，以适应 pipeline 变化。

它不是业务周期实例

Foundry 原生只知道 Dataset transaction、JobSpec logic 和 branch/scope；不天然知道 $bizdate、账期或同周期依赖。

Trigger Semantics

触发器语义

Time trigger 是瞬时条件

  • 按 cron 与 timezone 匹配。
  • 匹配时刻满足，过后不继续保持 satisfied。
  • DST 跳过的本地时间不会触发，重复的本地时间会触发两次。

Event trigger 会锁存

  • Data updated 表示 Dataset committed transaction。
  • 事件发生后保持 satisfied，直到整个 compound trigger 满足并 run。
  • 公开文档没有说明多个同类事件会按次数排队。

AND/OR 不是业务日期配对

AND(A updated, B updated) 只表示上次 run 后 A/B 都更新过，不表示 A/B 属于同一 business_date。

术语来源要分层

compound trigger、AND/OR trigger 与 event satisfied 语义来自 Palantir 官方文档；“状态机”“锁存”“不是业务时间 join”是本文的解释性抽象。

运行中触发会保持 triggered

前一次 schedule 仍在 action 时，新的触发会保持 triggered，等前一次 finished 后再运行；多个 pending trigger 是否合并，公开文档未说明。

Trigger Engine

多条件组合的内部判定模型

Palantir 没有公开 scheduler 源码；下面是由官方 trigger、schedule 和 troubleshooting 语义推导出的保守模型。

Trigger tree配置被表达成 time/event/AND/OR 的布尔树
Leaf statetime 是瞬时，event 是 observed latch
EvaluateAND 要全部 satisfied，OR 要任一 satisfied
Run gateroot satisfied 后发起 schedule run 或置 pending
Build dedupestaleness 决定是否真正创建 build
核心状态 trigger_tree + observed_events + last_run_time + in_action + pending_triggered
Event 语义 event happens -> leaf satisfied -> 保持到 whole trigger satisfied and schedule run

Advanced configuration 用 AND / OR 和括号组合 component triggers。

Schedule troubleshooting

官方证据：去重层

Fresh output 不重算；target 全部 up-to-date 时 schedule run 会被 ignored。

Builds core concepts Schedule troubleshooting

它如何判断组合条件

先更新叶子状态，再递归计算 trigger tree。Time leaf 只在 cron 命中瞬间为 true；Event leaf 在事件发生后锁存为 true；AND/OR 只做布尔组合。

它为何不等同时到达

AND(A,B) 判断的是上次 run 后 A/B 是否都 observed，而不是 A/B 是否同一秒到达，也不是同一业务日期。

运行中再次触发

官方只说明会保持 triggered，前一次结束后再 run；多次 pending 是否计数未公开，保守理解为至少一次 pending signal。

背后思考

Schedule 负责唤醒，Build/staleness 负责去重和成本控制。这套模型服务 freshness，不服务业务周期实例判重。

Build Resolution

Build、staleness 与 force build

Staleness 是数据版本判断

如果输入 Dataset 和 JobSpec logic 自上次 build 以来没有变化，输出被视为 fresh。fresh output 通常不重算。

Force build 只绕过 freshness

Force build 会绕过 up-to-date 判断，适合外部源不可见的 ingest 层；它不能替代业务日期补数。

Build range 会动态解析

Single、full/upstream、connecting、downstream 等范围都要结合 target、excluded datasets、branch、fallback 和 scope 来解析。

Incremental 追平是另一层

transaction limit 可能让一次成功 build 后 target 仍 stale；Re-trigger upon successful build 负责继续追平 backlog。

Data Connection

Data Connection sync 的特殊边界

外部源变化不进入内部 staleness

普通 Transform 的输入是 Foundry 内部 Dataset，平台能看到 transaction；Data Connection sync 的输入在外部，Foundry 不知道外部源是否已变化。

  • sync 层建议单独 schedule。
  • sync 层通常需要 force build。
  • downstream Transform 再使用普通 dataset update 或 freshness。

推荐分层

不要对整条 pipeline force build。只 force raw ingest/sync 层，下游继续由 Dataset transaction 和 staleness 驱动。

Architecture Decision

拆开业务周期调度与 freshness 调度

Business-cycle scheduler

负责业务周期是否 eligible

  • business_date / data_interval
  • ready manifest / watermark barrier
  • 补数、重跑、同周期依赖
Freshness scheduler

负责数据版本是否 stale

  • input transaction set
  • logic version / JobSpec
  • force build / external unknown

执行顺序

  1. ready manifest 判断同一 data interval 是否到齐。
  2. 创建 Run Identity。
  3. 解析 input transaction vector。
  4. 做 freshness/staleness 判断。
  5. build 成功后更新 partition manifest。

禁止混用

Data updated 事件只表示 transaction 变化，不表示业务日期完成。任何声明业务日期的输出，都不能只靠 Foundry-style trigger expression 驱动。

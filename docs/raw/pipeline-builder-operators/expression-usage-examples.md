# Expression 典型用法

## 背景

这篇文档的直接结论是：当前仓库已经拥有一份数量闭合的 expression 正式清单，但还缺一篇面向混合型读者的“类别入口 + 代表场景”主文档。`docs/raw/pipeline-builder-operators/artifacts/pb-expression-final/expression_inventory.json` 与 `docs/raw/pipeline-builder-operators/artifacts/pb-expression-final/expression_inventory_summary.json` 已经回答了“当前正式清单有多少条、数量是否闭合、哪些字段仍有自动解析局限”，却没有回答“遇到某类计算或结构处理问题时，通常应该先想到哪类 expression”。

因此，`Issue #4` 的职责不是再做一版 335 条清单复写，也不是把 expression 页面改写成参数字典，而是在既有正式清单之上补出一层更容易进入的组织方式。本文把已经冻结的 10 个高频场景放到同一张阅读地图里，让读者先知道常见问题分布在哪些类别，再进入后续场景章节继续看代表函数、适用边界和证据强度。

## 问题

当前 expression 侧最主要的缺口不是“有没有函数条目”，而是“怎么按问题理解这些条目”。正式清单已经闭合到 335 条，数量和 slug 唯一性也已经对齐 summary，但如果读者第一次进入这个主题，仍然很难快速判断空值兜底、条件分支、JSON 解析、结构体取值、数组判断这些常见问题分别落在哪些类别。

如果直接把 `Issue #2` 的正式清单当作 `Issue #4` 的答案，文档会退回成字段摘录；如果反过来把这篇文档写成大量函数参数说明，又会把“典型用法”误做成“参数手册”。这篇文档需要处理的正是中间这一层：它既要保留与正式清单一致的类别和证据边界，又要用场景化语言告诉读者哪些 expression 最值得先掌握、哪些例子只能按说明性方式表达、哪些结论还不能夸大成仓库内已复现实验结果。

## 目标

本文只服务一个核心目标：为 expression 建立一篇主文档级入口，让混合型读者能够用“问题场景”进入，用“代表函数与类别”继续下钻。这里的混合型读者既包括想判断能力范围的产品、分析和调研读者，也包括后续要继续实现、验证或迁移写法的开发读者。

围绕这个目标，本文第一版只做三件事。第一，把 expression 常见问题收敛为已经冻结的 10 个场景。第二，把这些场景与正式清单里的代表函数、类别标签和证据边界挂钩。第三，明确说明本文不是 335 条正式清单的复写，也不是参数字典，而是一篇帮助读者快速定位 expression 用法入口的类别型主文档。

## 场景 1：算术组合与派生指标（Numeric）

### 场景说明

这个场景要解决的问题不是“会不会做一次加法”，而是“怎样把多个数值字段组织成可解释的派生指标”。在实际表达式链路里，数值处理通常不会只停在单一步骤：读者往往先把若干字段组合成一个中间结果，再决定是否要做绝对值、差值、比例或阈值比较，最后把这个结果交给后续分支、排序或下游 schema。

对产品与调研读者来说，这一场景回答的是“哪些指标可以直接从已有数值列派生出来”；对实现与开发读者来说，更关键的是理解三类动作的先后关系。`addV2` 这类函数负责把多个数值输入收敛成一个可复用结果，`absV1` 这类函数负责把数值变换到更适合比较或展示的形态，`greaterThanV2` 这类比较函数则把派生结果继续推进到布尔判断。也就是说，数值组合、数值变换和阈值比较虽然属于不同动作，但在实际写法里经常是一条连续链路。

### 典型输入与输出

当前仓库缺少 expression 本地输入输出样例，因此这里的例子只能理解为说明性写法，而不是仓库内已复现实验。一个典型问题是：输入里有 `base_amount`、`adjustment_amount` 和 `target_amount` 三列，先把前两者组合成一个派生值，再判断它是否高于目标值。

按正式清单中的函数结构，可推知一种常见写法会先用 `addV2(base_amount, adjustment_amount)` 得到组合结果，再视需要用 `absV1(...)` 把正负方向抹平，最后用 `greaterThanV2(derived_amount, target_amount)` 产出布尔结果。这里的“输出”可能仍是数值，例如“校正后总额”；也可能已经进入比较阶段，变成一个布尔值，例如“是否超过阈值”。这一点很重要，因为它说明派生指标不总是最终列名，也可能只是后续判断链路里的中间节点。

### 关键写法

`addV2` 的关键性不在于“支持加法”本身，而在于它把多个 `Expression<Numeric>` 作为列表输入，适合表达“同一业务指标由多列共同构成”的场景。这样写的好处是，读者在阅读表达式时能直接看出“这是一个组合指标”，而不是把多次二元运算散落到不同地方。

`absV1` 之所以值得在这里一起讲，是因为很多派生指标并不关心方向，只关心偏离幅度。例如差额、波动量或修正值如果允许正负并存，直接比较可能会把“方向”与“幅度”混在一起；先取绝对值，才更容易表达“偏差是否超过容忍范围”。`greaterThanV2` 则承担另外一层职责：它本身不是 Numeric 类别，而是 Boolean 返回值，但经常是数值场景真正落地为业务规则的最后一步。换句话说，数值场景里的关键写法并不是某一个函数单独完成，而是“先组合、再变换、再比较”的连续结构。

### 适用时机与边界

当问题本质上是在问“已有数值列怎样合成一个更有业务含义的结果”时，这个场景最合适，例如总额、偏差、差值、风险分或阈值标记。它尤其适合作为后续条件分支的前置步骤，因为很多 `caseV2` 分支判断都依赖这里先产出的布尔或数值中间结果。

它的边界也要写清。第一，当前主文档不提供仓库内可复现的数值样例，因此不能把这里的说明性结果当成已验证口径。第二，`greaterThanV2` 属于比较函数而不是派生函数，只有在“派生结果要继续被判断”时才应并入这一场景。第三，正式清单能告诉我们函数签名和类别，却不能直接替代具体业务公式设计；“该不该取绝对值”“比较阈值放在哪一步”仍属于实现判断，而不是正式清单给出的事实。

### 当前证据

正式清单结构层面，`expression_inventory.json` 可以稳定定位 `addV2`、`absV1` 与 `greaterThanV2` 的 `slug`、`category`、`supported_in`、`arguments`、`return_type` 与 `source_url`。其中 `addV2` 的参数是 `List<Expression<Numeric>>`，`absV1` 的返回类型保留为类型变量 `T` 并在 `notes` 里注明 `T accepts Numeric`，`greaterThanV2` 则返回 `Boolean`。

summary 结果层面，`expression_inventory_summary.json` 已确认正式清单总量与 slug 唯一性闭合，但记录级 `evidence_type` 仍统一为 `线索`。可稳定定位的页面字段层面，我们当前能依赖的是标题、类别、支持环境、参数描述和返回类型这些结构化结果。基于这些材料做出的判断是：Numeric 场景不应只理解为算术函数列表，而应理解为“数值组合产出派生结果，并常常继续进入比较”的连续问题链。

## 场景 2：条件分支与布尔判断（Boolean / Popular）

### 场景说明

这个场景处理的是“什么时候只做真假判断，什么时候已经进入分支选择”。很多读者第一次看到这些函数时，会把 `caseV2`、`equalsV1`、`andV1` 都理解成“条件相关函数”，但它们解决的其实不是同一层问题。`equalsV1` 负责产出一个单点判断，`andV1` 负责把多个布尔结果合并成更严格的总体条件，而 `caseV2` 则是在已有条件基础上真正做出分支选择。

因此，这一场景最重要的不是记住三个 slug，而是分清“判断”和“分支”之间的层级。对产品与调研读者来说，这能帮助理解某条规则到底是在做命中判定，还是已经在做分类输出；对实现与开发读者来说，这能避免把所有复杂逻辑都塞进一个 `caseV2` 里，导致判断条件难以复用、也难以验证。

### 典型输入与输出

这里同样只能给说明性例子，而不能声称仓库内已有本地复现实验。一个常见问题是：当 `status` 等于 `ACTIVE`，且 `score` 高于某个阈值时，输出 `eligible`；否则输出 `ineligible`。按官方函数结构可推知，较清晰的链路通常不是直接把所有判断塞成一行，而是先用 `equalsV1(status, "ACTIVE")` 得到第一个布尔条件，再把它与另一个比较结果通过 `andV1(...)` 组合，最后交给 `caseV2` 决定输出值。

在这种写法里，`equalsV1` 和 `andV1` 的输出都是布尔值，`caseV2` 的输出则取决于分支值本身，可以是字符串、数值或其他类型。这个差别说明：布尔判断回答的是“条件是否成立”，而分支表达式回答的是“条件成立后返回什么”。如果把这两层混成同一种能力，就很难读懂表达式是在“筛选”还是在“赋值”。

### 关键写法

`equalsV1` 的关键性在于它把比较对象明确拆成 `Left` 与 `Right`，适合表达最基础、也最容易复用的业务判断，例如状态是否相等、编码是否匹配、字段是否命中指定值。它通常是分支逻辑的原子条件，而不是分支本身。

`andV1` 关键在于它接收的是 `List<Expression<Boolean>>`。这意味着它不是做二选一输出，而是在已有多个布尔判断之间收紧条件边界。对于实现者来说，这比把复杂逻辑硬写进一个嵌套分支更可读，因为每个条件可以先作为独立判断存在，再组合成“全部满足才继续”的结果。

`caseV2` 则是这一场景里真正承担 if/else 语义的代表函数。它的 `Branches` 参数是由“条件 + 返回值”组成的列表，`Default` 则定义全部条件不命中时的兜底结果。也正因为如此，`caseV2` 与前两个函数的职责不同：前两个函数在生产布尔条件，`caseV2` 在消费这些条件并选择最终输出。把这层区别写清，比单独列出它们属于 Boolean 或 Popular 更重要。

### 适用时机与边界

当问题是在问“某个条件是否成立”时，应优先从 `equalsV1`、`greaterThanV2` 这类单点判断出发；当问题是在问“多个条件要不要同时满足”时，应引入 `andV1` 这类布尔组合；当问题已经变成“不同条件命中后分别返回什么”时，才真正进入 `caseV2` 的场景。这个顺序能帮助读者把表达式拆成更稳定的层级，而不是把判断与分支混为一谈。

边界上需要特别提醒两点。第一，`caseV2` 虽然被归在 `Popular` 类别，但它并不替代布尔函数本身；它的前提仍是你已经有可读、可解释的条件表达式。第二，正式清单提供的是结构信息，不包含本地验证过的复杂嵌套分支样例，因此这里的建议属于基于函数结构做出的稳定判断，而不是对某种具体业务规则写法的唯一答案。

### 当前证据

正式清单结构层面，`caseV2`、`equalsV1` 与 `andV1` 都能在 `expression_inventory.json` 中稳定定位。`caseV2` 记录显示其类别为 `Popular`，参数由可选的 `Default` 与 `List<Tuple<Expression<Boolean>, Expression<T>>>` 的 `Branches` 组成；`equalsV1` 的左右输入都是 `Expression<ComparableType>`；`andV1` 的输入则是布尔条件列表，返回值为 `Boolean`。

summary 结果层面，当前正式清单在总量和唯一性上已经闭合，但这三条记录的 `evidence_type` 也仍是 `线索`。可稳定定位的页面字段层面，我们能够直接引用的是标题、类别、参数结构、返回类型和支持环境。基于这些材料做出的判断是：`equalsV1` 解决单点比较，`andV1` 解决布尔组合，`caseV2` 解决分支选择，它们应该按条件链路而不是按“都和条件有关”这一层混写。

## 场景 3：空值兜底与默认值（Data preparation）

### 场景说明

这个场景回答的是：当字段可能缺失、可能是真正的 `null`，也可能只是空字符串时，表达式应该怎样保持结果稳定。很多数据准备问题并不是“值错了”，而是“某些记录没有值，或者看起来有值但实际上只是空白占位”。如果不先处理这层不稳定性，后续的字符串、数值或日期计算都会被连带放大。

这里要特别区分两类动作。`isNullV1` 与 `isNotNullV1` 负责判断当前字段处在什么状态，`firstNonNullV1` 则负责在多个候选值之间做兜底选择。换句话说，空值判断回答的是“现在有没有值”，兜底函数回答的是“如果没有值，该退到哪一个候选”。这两步经常一起出现，但不是同一个动作。

### 典型输入与输出

当前仓库没有 expression 的本地样例，因此下面仍是说明性例子。假设输入里有 `primary_email`、`backup_email` 与 `fallback_label` 三个字段，其中前两个可能为 `null`，也可能是空字符串。按正式函数结构可推知，一种典型写法会先决定是否把空字符串也视为缺失值，再用 `firstNonNullV1` 依次挑选第一个可用结果。

例如，若把 `Treat empty strings as null` 设为真，`firstNonNullV1(primary_email, backup_email, fallback_label)` 的说明性输出就不再把 `""` 视作“已经有值”，而是继续向后寻找下一个候选。与之配合时，`isNullV1` 或 `isNotNullV1` 可以用来单独产出一个布尔列，标记某个原始字段是否真正可用。这说明“兜底值”与“空值标记”往往同时需要，一个服务于最终取值，一个服务于状态解释。

### 关键写法

`firstNonNullV1` 的关键点不只是 coalesce 语义，而是它把“候选值顺序”显式写进表达式里。对于开发读者来说，这意味着默认值策略不是散落在多层 if/else 里，而是以一个清晰的优先级链存在：先看主字段，再看备选字段，最后退到默认常量或说明字段。

这里最值得单独强调的是 `Treat empty strings as null`。如果不启用这层语义，空字符串会被当作一个已经存在的值，`firstNonNullV1` 就可能在第一个候选位置提前停止，导致后续真正有业务意义的备选值永远没有机会被取到。`isNullV1` 与 `isNotNullV1` 同样带有这一参数，也说明在官方函数结构里，“空字符串要不要等同于缺失值”不是旁支细节，而是空值场景的核心边界之一。

### 适用时机与边界

当输入字段来自多源拼接、人工录入、半结构化解析或历史 schema 迁移时，这个场景最常见。它适合作为许多后续场景的前置步骤，因为字符串清洗、日期解析和条件分支都很依赖“值是否真的存在”这件事先被稳定定义。

边界上需要保留两层谨慎。第一，`firstNonNullV1` 只能解决“候选值谁先取”的问题，不能自动判断哪个值在业务上更可信，候选顺序仍要由实现者决定。第二，把空字符串视为 `null` 虽然常常更符合数据准备直觉，但并不适用于所有字段；如果空字符串本身有业务语义，就不能无条件开启这一参数。

### 当前证据

正式清单结构层面，`firstNonNullV1` 在 `expression_inventory.json` 中的参数包含 `Expressions` 与 `Treat empty strings as null`，返回类型为类型变量 `T`，并在 `notes` 里注明 `T accepts AnyType`。`isNullV1` 与 `isNotNullV1` 也都明确列出了待检查表达式和同名的“是否将空字符串视作 null”参数，返回类型均为 `Boolean`。

summary 结果层面，正式清单的总量、唯一性和异常检测范围已经固定，但这些记录目前仍统一属于 `线索`。可稳定定位的页面字段层面，我们能直接引用的是函数标题、类别、参数名与参数描述。基于这些材料做出的判断是：空值场景不能只写“有个 coalesce 函数”，而应把“判断当前是否为空”“怎样决定兜底顺序”“是否把空字符串视为缺失值”作为同一条问题链一起讲清。

## 场景 4：字符串清洗、匹配与拆分（String / Data preparation / Regex）

### 场景说明

这个场景最容易被误写成“四个彼此无关的字符串函数”，但读者真正需要的其实是一条问题链：原始字符串往往先要清洗，再决定是否命中某个模式，随后可能要做局部替换，最后才拆成更适合下游处理的数组或片段。把这几步拆开讲可以理解函数；把它们连成链讲，才更像真实的数据准备场景。

因此，这里不把 `cleanStringV1`、`stringContainsV1`、`regexReplaceV1`、`splitStringV2` 看成平行散点，而是把它们放进同一条处理路径里。对产品与调研读者来说，这能回答“脏字符串通常怎样被整理成可判断、可拆分的结果”；对实现与开发读者来说，则能帮助区分哪些步骤在做标准化，哪些步骤在做命中判断，哪些步骤已经在重写或重组字符串结构。

### 典型输入与输出

这里的输入输出仍然只是说明性例子。假设某个输入字段包含人工录入的标签串，例如前后有多余空白、内部分隔符不稳定，还夹杂需要统一替换的模式。按正式函数结构可推知，一条典型链路会先用 `cleanStringV1` 做裁剪、空白标准化或空串归零，再用 `stringContainsV1` 判断它是否包含某个关键词，必要时用 `regexReplaceV1` 做模式替换，最后用 `splitStringV2` 按分隔规则拆成数组。

这种链路的说明性输出不是单一类型。清洗和替换阶段的输出仍是字符串，包含判断阶段输出变成布尔值，拆分阶段输出则是 `Array<String>`。这恰好说明字符串场景经常横跨清洗、判断和结构化三个层级，而不是“字符串进，字符串出”这么简单。

### 关键写法

`cleanStringV1` 的关键性在于它把一组清洗动作作为显式参数暴露出来，其中包括 `Normalize whitespace`、`Nullify empty` 与 `Trim`。这意味着“清洗”不是一个模糊动作，而是可以清楚表达究竟做了哪些标准化步骤。对数据准备读者来说，尤其重要的是 `Nullify empty`，因为它常常决定后续场景是继续走字符串处理，还是转入空值兜底。

`stringContainsV1` 的关键点不只是 contains 语义，而是它同时把待检查值、比较目标和值是否忽略大小写都写成显式输入。这让“命中判断”不再依赖隐式约定，尤其适合在字符串已经过基本清洗后，快速形成一个布尔信号。

`regexReplaceV1` 和 `splitStringV2` 则进一步把“替换”和“拆分”从简单文本处理推进到结构调整。前者适合处理规则更复杂的局部模式修正，后者直接返回数组，并通过 `Pattern` 与 `Limit` 定义拆分边界。这里的关键不是记住正则参数名，而是理解为什么真实链路常常先做清洗和替换，再做拆分，否则脏分隔符与不稳定空白会直接污染后续数组结构。

### 适用时机与边界

当原始值来自表单输入、日志片段、标签串、半结构化文本或多系统拼接字段时，这个场景最常见。它适合作为进入数组处理、条件判断或 map 构造之前的准备步骤，因为这些下游动作通常都要求字符串先变得可预测。

边界上要避免两种误读。第一，`stringContainsV1` 虽然也属于字符串相关场景，但它返回的是布尔值，因此它解决的是“是否命中”而不是“如何清洗”。第二，`regexReplaceV1` 与 `splitStringV2` 都引入模式语义，正式清单能支持我们确认它们的参数结构，却不能替代具体正则设计与样例验证；复杂模式仍需要后续独立验证，不能在本文里暗示已经完成本地实验。

### 当前证据

正式清单结构层面，`cleanStringV1` 的类别同时覆盖 `Data preparation` 与 `String`，参数中能稳定看到清洗动作集合与输入字符串；`stringContainsV1` 既出现在 `String` 也出现在 `Boolean` 类别，返回 `Boolean`；`regexReplaceV1` 位于 `Regex` 与 `String` 类别；`splitStringV2` 返回 `Array<String>`，并带有 `Pattern` 与 `Limit` 参数。

summary 结果层面，当前正式清单已完成 335 条闭合，但这些记录仍统一记为 `线索`。可稳定定位的页面字段层面，我们能够直接引用的是各函数的类别组合、参数描述、返回类型与官方来源链接。基于这些材料做出的判断是：字符串场景最适合按“清洗 -> 命中判断 -> 模式替换 -> 拆分重组”的问题链组织，而不是退回成四个散点函数的堆叠。

## 场景 5：日期时间解析、格式化与区间计算（Datetime / Cast）

### 场景说明

这个场景要回答的是：原始时间值经常先以字符串形式出现，但真正的时间分析往往需要先把它解析成时间对象，再做区间计算、提取日期部分或进入后续格式化链路。也就是说，日期时间问题通常不是单个函数就能结束，而是由“解析 -> 计算 -> 提取或展示”几步组成。

在这条链路里，`stringToTimestampV2` 解决的是入口问题：字符串怎样进入可计算的时间类型；`timestampDiffV1` 解决的是区间问题：两个时间点之间相差多少；`datePartV1` 解决的是拆解问题：已经得到的日期或时间对象，怎样再取出年、月、周、日等维度；`timestampToStringV2` 则承担展示或落盘前的格式化动作。这几个步骤分别服务不同阶段，混在一起写时最容易忽略的，就是格式与时区对整个链路的影响。

### 典型输入与输出

这里的输入输出同样只能按说明性例子表达。假设输入里有字符串列 `event_time_text`，格式可能接近 ISO8601，也可能需要额外提供格式模板与时区。按正式清单中的参数结构，可推知典型链路会先用 `stringToTimestampV2` 把字符串解析成 `Timestamp`，再用 `timestampDiffV1(end_ts, start_ts, unit)` 计算间隔，视问题需要用 `datePartV1(parsed_ts, Month)` 之类写法提取月份、年份或周次，最后也可能通过 `timestampToStringV2(parsed_ts, format, timezone)` 重新格式化成对下游更友好的字符串。

因此，这个场景里的输出可能有四种形态：解析阶段输出 `Timestamp`，区间计算阶段输出 `Long`，日期部分提取阶段输出 `Integer`，格式化阶段则重新返回 `String`。这也说明所谓“日期时间场景”并不只是在做格式转换，而是在把原始字符串推进到可比较、可聚合、可分组的时间结构里，再在需要时重新回到展示友好的文本形态。

### 关键写法

`stringToTimestampV2` 的关键点在于它不只是接受一个字符串输入，还显式暴露了格式列表与时区参数。这里最值得强调的是：如果原始字符串本身不带时区，`Time zone` 参数就直接参与解析；如果原始格式里已经带了时区，正式清单中的参数描述也提醒了这一参数可能覆盖原值。也就是说，时区不是附属细节，而是决定解析结果能否稳定对齐的边界点。

`timestampDiffV1` 之所以关键，在于它把结束时间、开始时间和单位拆成三个显式参数，提醒读者“区间计算”不仅是相减，还包含单位选择这一层语义。天、小时、周、月得到的结果并不只是同一个数字换个展示方式，而是可能服务完全不同的业务口径。

`datePartV1` 则承担把时间对象重新拆回业务维度的作用。很多下游报表、聚合或分组并不直接使用完整时间戳，而是使用年、季度、周次或日。`timestampToStringV2` 进一步承担展示或导出前的格式化动作，它把时间对象重新转成字符串，并显式暴露 `Format` 与 `Time zone` 参数。把它们和解析、区间计算放在同一场景里，是为了说明日期时间表达式常常是一条完整加工链，而不是孤立地“解析一个时间”。

### 适用时机与边界

当原始数据以字符串提供时间值，而下游需要做时长计算、时间窗口判断、按日期维度分组或生成更稳定的时间字段时，这个场景最合适。它既是时间分析的入口，也是很多窗口、聚合和过滤条件的前置准备。

边界上需要比其他场景更谨慎。第一，格式模板和时区决定了解析结果是否可靠，如果输入格式本身不稳定，`stringToTimestampV2` 就不能被当作“自动兜底一切”的入口。第二，`timestampDiffV1` 对单位很敏感，同一对时间点用不同单位得到的数值不可直接混用。第三，`timestampToStringV2` 虽然能承担格式化，但它描述的是“怎样输出一个已经得到的时间对象”，并不能反向证明输入字符串一定能被正确解析。第四，当前主文档没有本地时间样例，因此这里不能把任何说明性结果写成已经验证过的时区行为或格式兼容结论。

### 当前证据

正式清单结构层面，`stringToTimestampV2` 在 `expression_inventory.json` 中同时归到 `Cast` 与 `Datetime` 类别，返回类型为 `Timestamp`，参数里能稳定看到字符串输入、格式列表与时区。`timestampDiffV1` 的参数包括 `End`、`Start` 与单位枚举，返回类型为 `Long`。`datePartV1` 接收 `Expression<Date | Timestamp>` 与日期部分枚举，返回 `Integer`。`timestampToStringV2` 则归到 `Cast`、`Datetime` 与 `String` 类别，返回 `String`，并明确给出 `Timestamp`、`Format` 与 `Time zone` 参数。

summary 结果层面，正式清单的数量、slug 唯一性与异常检测范围已在 `expression_inventory_summary.json` 闭合，但这些记录的 `evidence_type` 仍为 `线索`。可稳定定位的页面字段层面，我们目前可以直接引用函数标题、类别、参数描述、返回类型和官方链接。基于这些材料做出的判断是：日期时间场景应按“先解析成时间对象，再计算区间，再提取日期部分，必要时重新格式化输出”的链路理解，而格式与时区是这条链路最需要显式保留的边界。

## 场景 6：类型转换与 schema 边界适配（Cast）

### 场景说明

这个场景要解决的问题不是“会不会做一次 cast”，而是“表达式链路什么时候需要显式跨过类型边界，什么时候又是在保持物理类型不变的前提下补 logical type 约束”。很多读者第一次看到 `castV2` 和 `logicalTypeCastV1` 时，会把两者都理解成“转换类型”，但它们处理的其实是不同层级的 schema 适配问题。

`castV2` 更像是在处理物理类型或返回类型层级的边界。例如某个上游字段当前被当作 `AnyType`、字符串或其他不够稳定的表达式结果，但下游数值、时间或结构化写法要求它先被明确 cast 到某个目标类型，这时解决的是“表达式怎样进入一个新类型”。`logicalTypeCastV1` 则不是去改写底层值的物理类型，而是在同一物理承载上补一个逻辑类型（logical type）约束，并允许在约束不成立时回退到默认值。对产品与调研读者来说，这一差别对应的是“字段类型改写”和“字段语义约束”不是一回事；对实现与开发读者来说，这关系到 schema 边界到底落在值本身，还是落在值要满足的业务约束上。

### 典型输入与输出

当前仓库缺少 expression 本地输入输出样例，因此这里仍然只能给说明性例子。一个常见问题是：上游输入里某个字段目前只是“可读但类型不稳定”的值，下游却要求它进入某个明确 schema 字段。例如某列先以通用表达式形式存在，但后续数值计算或结构化字段写入要求它先变成明确的 `Type<C>`，这类链路会先通过 `castV2(expression, target_type)` 把值推进到目标类型。

另一类问题则更接近逻辑约束适配。假设某个值在物理层面已经可以保留原有承载方式，但下游 schema 希望它满足特定逻辑类型限制，例如满足某种格式、业务语义或可接受范围，那么说明性写法会更接近 `logicalTypeCastV1(expression, logical_type, optional_default)`。这里的输出并不是“换了一种底层存储”，而是“同一类值被重新声明为符合某个逻辑类型；若不符合，则退回默认值或 null”。这说明类型转换场景不应只被理解为“把字符串转数字”，而要理解成“让值能穿过下游 schema 边界”。

### 关键写法

`castV2` 的关键职责，是把一个当前类型不够明确或不适配下游写法的表达式，显式转换成目标 `Type<C>`。它的重要性不在于“支持 cast”本身，而在于它让 schema 边界不再是隐式猜测。只要下游函数、结构字段或输出模式已经期待某种明确返回类型，`castV2` 就是在链路里补上这一层类型承诺。

`logicalTypeCastV1` 则承担另一层职责：它处理的不是物理类型替换，而是逻辑类型约束。正式清单摘要里已经明确提示，“Unlike the regular cast expression, this expression will not change...” 这一差异可以稳定支持我们作出如下判断：它更像是在既有值之上施加语义边界，而不是把值重写成另一类底层类型。它的 `optional Default value` 之所以关键，是因为逻辑约束不像普通 cast 那样只有“转得过去或转不过去”两种状态；很多场景里，更重要的是当值不满足目标 logical type 时，下游是否还能拿到一个可解释的回退结果。

因此，这一场景的代表写法不是把两个函数并列成“都是 cast”，而是把它们放进同一条边界链路里理解：`castV2` 负责解决“值如何进入目标类型”，`logicalTypeCastV1` 负责解决“值如何在不改写底层类型的前提下满足目标逻辑语义”。前者更接近类型落盘或函数入参适配，后者更接近 schema 语义校正。

### 适用时机与边界

当问题是在问“上游值为什么还不能直接接到下游 expression 或 schema 字段上”时，这个场景最合适。它尤其常见于字符串、数值、时间和复杂结构之间的衔接步骤，也常出现在 JSON 解析、结构体组装或 map/value 读取之后，因为这些结果虽然已经可用，但未必自动符合下游期望的明确类型或逻辑类型。

边界上需要保留三层谨慎。第一，`castV2` 解决的是类型适配，不负责证明源值一定语义正确；把值 cast 成目标类型，并不等于业务意义自动成立。第二，`logicalTypeCastV1` 解决的是逻辑约束适配，不等于底层值真的被重编码；因此它和普通 cast 不能互相替代。第三，当前主文档没有本地输入输出样例，所以这里不能把任何说明性示例写成“仓库已验证某 logical type 行为”的结果，只能把它们写成基于正式清单参数结构与摘要信息的稳定判断。

### 当前证据

正式清单结构层面，`expression_inventory.json` 可以稳定定位 `castV2` 与 `logicalTypeCastV1` 的 `slug`、`category`、`supported_in`、`arguments`、`return_type` 与 `source_url`。其中 `castV2` 的参数是 `Expression<AnyType>` 与 `Type<C>`，返回类型是 `C`；`logicalTypeCastV1` 的参数则是 `Expression<C>`、`Logical type` 与可选默认值，返回类型同样保留为 `C`。

summary 结果层面，`expression_inventory_summary.json` 已确认正式清单总量闭合，但这两条记录的 `evidence_type` 仍统一为 `线索`。可稳定定位的页面字段层面，我们当前能直接依赖的是类别标签、参数结构、返回类型和英文摘要。基于这些材料做出的判断是：类型转换场景不应只理解为普通 cast，而应理解为“物理类型适配”和“逻辑类型约束”两层 schema 边界管理。

## 场景 7：数组成员判断与数组重组（Array）

### 场景说明

这个场景处理的不是“数组里有哪些函数”，而是“一个集合型字段怎样先被检查、再被整理、最后再衔接到下游表达式”。真实链路里，数组往往不会只做一个动作就结束。很多问题先要回答“某个成员是否存在”，接着才会进入“要不要去重、怎样压平或拼接”，最后再决定结果是继续保留数组，还是转成字符串或其他下游更好消费的形态。

因此，这一场景最适合按“集合检查 -> 结构整理 -> 下游衔接”的顺序来理解。`arrayContainsV1` 负责集合检查，回答的是“数组里有没有这个成员”；`arrayDistinctV1` 负责结构整理，回答的是“重复项要不要先被去掉”；`arrayJoinV1` 则更接近下游衔接，回答的是“数组是否要被重组为一个可展示、可比较或可落盘的字符串结果”。对产品与调研读者来说，这能帮助理解数组处理不是只有 membership 判断；对实现与开发读者来说，这说明数组场景里的关键是步骤次序，而不是某个函数单独有多复杂。

### 典型输入与输出

这里的输入输出同样只能作为说明性例子。假设某个输入字段是标签数组 `["risk", "vip", "risk", "manual"]`，业务首先要判断是否包含 `vip`，再把重复标签去掉，最后把整理后的标签串传给下游展示或规则匹配。这种链路里，说明性写法通常会先用 `arrayContainsV1(tags, "vip")` 产出布尔值，再用 `arrayDistinctV1(tags)` 得到去重后的数组，最后用 `arrayJoinV1(distinct_tags, ",")` 产出拼接后的字符串。

这个场景里的输出会跨过三种类型。集合检查阶段返回 `Boolean`，去重阶段仍返回 `Array<T>`，拼接阶段则返回 `String`。这很重要，因为它提醒读者：数组问题并不总以数组结束。有些链路只是借助数组中转，真正的目标可能是产出一个布尔标记、一个干净数组，或者一个可读字符串。

### 关键写法

`arrayContainsV1` 的关键职责，是把“集合里是否存在指定成员”显式写成一个布尔判断。它适合承担很多规则的入口条件，例如标签命中、白名单成员存在、权限组是否包含某项。它之所以要放在链路最前面讲，是因为 membership 判断常常决定后面是否还值得继续处理数组本身。

`arrayDistinctV1` 则承担结构整理职责。它的重要性不只是“去重”，而是帮助数组回到更稳定的集合语义。如果上游数组来自字符串拆分、多源拼接或重复累积，直接拿去拼接或比较，会把重复项带到下游；先做 distinct，才能让后续判断、展示和 map 构造更可解释。

`arrayJoinV1` 是这个场景里最容易被低估的一步。它不是简单把数组变成字符串，而是在做“从集合形态衔接到下游消费形态”的转换。当下游只接受单个字符串字段、展示文本、日志摘要或进一步的字符串匹配时，`arrayJoinV1` 才是把数组结果真正接到下一段链路上的动作。也正因为这样，这一场景的关键写法不是单独记住三个函数，而是理解它们分别位于集合检查、结构整理和下游衔接的三个位置。

### 适用时机与边界

当字段天然就是多值集合，例如标签、角色列表、命中的规则集合或从字符串拆出来的片段数组时，这个场景最合适。它尤其适合作为字符串清洗和拆分之后的下一步，因为数组常常是从 `splitStringV2` 这类函数进入表达式链路的。

边界上需要提醒两点。第一，`arrayContainsV1` 返回的是布尔值，它解决的是命中判断，不负责整理数组结构。第二，`arrayJoinV1` 会把数组压回字符串，因此一旦需要保留成员级别的可继续处理能力，就不应过早 join。换句话说，join 是一种下游衔接选择，而不是所有数组场景的默认终点。

### 当前证据

正式清单结构层面，`arrayContainsV1`、`arrayDistinctV1` 与 `arrayJoinV1` 都能在 `expression_inventory.json` 中稳定定位。`arrayContainsV1` 归于 `Array` 与 `Boolean` 类别，输入是 `Expression<Array<ComparableType>>` 与待查值，返回 `Boolean`；`arrayDistinctV1` 接收 `Expression<Array<T>>` 并返回 `Array<T>`；`arrayJoinV1` 则接收 `Expression<Array<String>>` 与分隔符，返回 `String`。

summary 结果层面，正式清单在总量与唯一性上已经闭合，但这三条记录的 `evidence_type` 仍统一为 `线索`。可稳定定位的页面字段层面，我们能直接引用的是类别组合、参数结构、返回类型与官方链接。基于这些材料做出的判断是：数组场景最适合按“先判断成员、再整理集合、最后决定怎样衔接下游”的问题链组织，而不是写成三个平铺的数组函数条目。

## 场景 8：结构体创建、取值与字段更新（Struct）

### 场景说明

这个场景最重要的前提是：struct 不是 JSON。它不是一段待解析的原始字符串，而是已经进入 expression 类型系统里的结构化对象。也正因为如此，struct 场景解决的问题，不再是“怎么把文本读懂”，而是“已经有了一个结构化对象后，怎样创建它、读取它、再局部更新它”。

这三类动作分别由三个代表函数承担。`createStructV2` 负责创建，回答的是“多个列怎样组合成一个结构化对象”；`getStructFieldV2` 负责读取，回答的是“怎样从 struct 里按 locator 取出目标字段”；`addOrUpdateStructFieldV1` 负责更新，回答的是“怎样在已有 struct 上新增一个字段，或覆盖原字段值”。对产品与调研读者来说，这能帮助区分“对象结构已经存在”和“对象还只是 JSON 字符串”两种完全不同的问题；对实现与开发读者来说，这意味着 struct 场景应按对象生命周期动作来理解，而不是按函数名散列记忆。

### 典型输入与输出

这里同样只能给说明性例子。假设输入里有 `name`、`email` 和 `country` 三个标量字段，希望先把它们组装成一个用户对象，再读取其中的 `email`，最后补上一层 `status` 字段。按正式清单结构，可推知一条典型链路会先用 `createStructV2(...)` 组合出一个 `Struct`，再通过 `getStructFieldV2(locator, struct_expr)` 读取某个字段，最后用 `addOrUpdateStructFieldV1(new_value, locator, struct_expr)` 返回更新后的新 `Struct`。

这一链路的输出会不断切换在“结构体整体”和“结构体内某个字段值”之间。创建和更新阶段的输出都是 `Struct`，读取阶段的输出则是 `AnyType`。这说明 struct 处理并不是“只要进了对象就一直在对象里”；很多链路会在对象整体和对象内部字段值之间来回切换，直到下游拿到真正需要的那一层结果。

### 关键写法

`createStructV2` 的关键职责，是把多个离散列明确收束为一个 `Struct`。它的重要性不在于“能打包多个字段”，而在于它让后续链路可以把这些字段当作一个对象整体继续传递，而不是始终以平铺列的形式散落在表达式中。

`getStructFieldV2` 的核心是 `StructLocator`。它意味着读取动作不是靠列名猜测，而是靠一个明确的路径去定位内部字段；正式清单里的参数描述还提示 locator 可以是多级路径，例如 `['author', 'email']`。这让 struct 读取具备了“沿结构路径深入”的语义，而不只是一级字段访问。

`addOrUpdateStructFieldV1` 则承担局部更新职责。它不是重新构造整份对象，而是在已有 `Struct` 上按 locator 增加或覆盖一个字段值。把它和 `createStructV2` 放在一起看，能更清楚地说明 struct 场景里至少有三种不同动作：第一次创建对象、按路径读取对象内部值、在对象上做局部修改。也正因为如此，本节最关键的不是“Struct 类里有三个函数”，而是“结构体对象的创建、读取、更新分别由不同写法承担”。

### 适用时机与边界

当问题已经进入嵌套对象建模，例如要把多列整理成一个复合字段、要从对象字段中继续取值、或要在已有对象上补一个新字段时，这个场景最合适。它也常作为 JSON 解析之后的下一步，因为很多链路会先把 JSON 字符串解析成 `Struct`，再进入本节的读取与更新动作。

边界上需要明确两点。第一，struct 不是 JSON 字符串；如果输入当前仍是一段原始 JSON 文本，应先进入场景 9，而不是直接拿 struct 函数处理。第二，`getStructFieldV2` 返回 `AnyType`，说明读取动作本身不会自动替你定义更具体的业务类型；如果下游要求更明确的类型或 logical type，往往还要与场景 6 的 cast 写法衔接。

### 当前证据

正式清单结构层面，`createStructV2`、`getStructFieldV2` 与 `addOrUpdateStructFieldV1` 都能在 `expression_inventory.json` 中稳定定位。`createStructV2` 的参数是 `List<Expression<AnyType>>`，返回 `Struct`；`getStructFieldV2` 的参数包含 `StructLocator` 与 `Expression<Struct>`，返回 `AnyType`；`addOrUpdateStructFieldV1` 的参数则包含新值表达式、`StructLocator` 与目标 `Struct`，返回更新后的 `Struct`。

summary 结果层面，正式清单的数量与唯一性已闭合，但这三条记录仍统一记为 `线索`。可稳定定位的页面字段层面，我们当前能直接引用的是标题、类别、参数结构、locator 描述、返回类型和官方链接。基于这些材料做出的判断是：struct 场景应按“创建对象 -> 读取字段 -> 局部更新”三种动作理解，并且要明确把 struct 与原始 JSON 文本区分开。

## 场景 9：JSON 字符串解析为结构化对象（Struct / Data preparation / Popular）

### 场景说明

这个场景回答的是：一段原始 JSON 字符串，怎样变成 expression 能继续读取、更新和组合的结构化对象。它之所以单独成节，是因为 JSON 文本与 struct 对象之间隔着一道非常关键的边界。只有先跨过这一步，场景 8 中的 struct 创建、取值和字段更新才真正有了可操作对象。

`parseJsonAsSchemaV3` 的关键不只是“能解析 JSON”，而是它要求调用者显式给出 `Schema` 和 `Output mode`。这两个参数决定了表达式到底是在“按什么结构读这段 JSON”，以及“解析失败时是静默落 null，还是把错误与成功结果一起保留下来”。对产品与调研读者来说，这说明 JSON 解析不是单纯的字符串转对象，而是带着目标结构假设进入结构化世界；对实现与开发读者来说，这也意味着 schema 设计和错误输出模式，直接决定了解析结果后续还能不能被安全接到 struct 链路。

### 典型输入与输出

这里的输入输出仍然只能写成说明性例子。假设某列 `payload_json` 存放的是 JSON 字符串，业务希望从中提取 `author.email`、`status` 和若干明细字段，并继续做字段更新或 map 构造。按正式清单结构，可推知典型写法会先用 `parseJsonAsSchemaV3(payload_json, schema_definition, output_mode)` 把字符串解析成目标结构。

这里最关键的是 schema 与输出模式的差异。若 `Schema` 定义为 `Struct`，输出就可以直接衔接到 `getStructFieldV2` 或 `addOrUpdateStructFieldV1`。若 `Output mode` 选 `simple`，无法按 schema 成功解析的字段会被当作 `null`；若选 `with errors`，结果会返回一个仍可解析的 struct，其中 `ok` 字段放有效解析结果，`error` 字段保留错误信息。也就是说，这个场景的“输出”并不是唯一形态，而是取决于你是否需要保留解析失败痕迹，以及希望下游接到的是干净对象，还是“对象 + 错误上下文”的复合结构。

### 关键写法

`parseJsonAsSchemaV3` 的第一个关键职责，是把 schema 作为显式输入，而不是假定 JSON 自己会长成什么样。正式清单中 `Schema` 的类型允许是 `Array<AnyType> | Map<String, String> | Struct`，这意味着解析结果的目标形态本身就是调用时要先决定的事情。对实现者来说，这一点非常重要，因为它决定下游究竟是进入数组场景、map 场景，还是 struct 场景。

第二个关键职责，是通过 `Output mode` 决定错误怎样被表达。`simple` 更像是把 JSON 解析视作数据准备步骤，只要字段解析失败就直接落成 `null`；`with errors` 则把错误本身也纳入结构化输出。它之所以关键，不是因为“多了一个报错模式”，而是因为这会直接影响下游链路设计。若后续要继续做字段追责、质量分析或差异排查，`with errors` 会更合适；若下游只关心尽快得到一个尽可能可用的对象，`simple` 会更接近清洗式入口。

第三个关键职责，是它与场景 8 的 struct 操作天然衔接。只要 `Schema` 定义出的目标是 `Struct`，`parseJsonAsSchemaV3` 就是从 JSON 文本世界进入 struct 世界的门；随后 `getStructFieldV2` 可以读取字段，`addOrUpdateStructFieldV1` 可以继续补字段，必要时还可以与场景 6 的 cast 配合，让某些取出的值再进入更明确的类型边界。

### 适用时机与边界

当原始输入首先是一段 JSON 文本，而下游目标是结构化取值、字段更新、复杂对象拼接或更稳定 schema 输出时，这个场景最合适。它尤其适合作为半结构化数据进入表达式链路的第一步，因为很多后续场景都默认你已经有了 `Struct`、`Map` 或 `Array`，而不是一段未解析字符串。

边界上要比普通字符串清洗更谨慎。第一，`parseJsonAsSchemaV3` 的价值依赖于 schema 是否足够明确；如果 schema 本身就不稳定，解析输出也会跟着摇摆。第二，`simple` 与 `with errors` 不是展示选项，而是错误语义选择；它们会直接影响下游是否还能区分“字段本来就是 null”和“字段解析失败后被置为 null”。第三，当前仓库没有本地 JSON 输入输出样例，因此这里不能把任何说明性例子写成已验证过的解析兼容性结果。

### 当前证据

正式清单结构层面，`parseJsonAsSchemaV3` 在 `expression_inventory.json` 中可稳定定位到 `Data preparation`、`Popular`、`String` 与 `Struct` 四类标签，参数包含 `JSON`、`Schema` 和 `Output mode`，返回类型为 `Array<AnyType> | Map<String, String> | Struct`。其中 `Output mode` 的参数描述已经明确区分 `simple` 与 `with errors` 的行为差异。

summary 结果层面，正式清单数量与唯一性已经闭合，但该记录的 `evidence_type` 仍是 `线索`。可稳定定位的页面字段层面，我们当前可以直接依赖的是类别组合、schema 参数类型、输出模式说明、返回类型和官方链接。基于这些材料做出的判断是：`parseJsonAsSchemaV3` 之所以关键，不只是因为它能解析 JSON，而是因为它决定了 JSON 文本怎样进入 struct、map 或数组世界，并通过 output mode 决定错误信息是否继续可见。

## 场景 10：map 构造与按 key 取值（Map）

### 场景说明

这个场景要解决的问题是：当数据更适合被表达成“key -> value”而不是“固定字段名 -> 字段值”时，expression 怎样创建、读取并局部更新一个 map。这里最需要先讲清的一点，是 map 访问和 struct 字段访问不是同一回事。struct 更像一份字段名事先固定、由 schema 定义边界的对象；map 则更像一个运行时按 key 查值的键值容器，重点不在固定字段路径，而在“给定 key 时能拿到哪个 value”。

因此，这一场景最适合按“构造 -> 按 key 取值 -> 局部更新”的链路理解。`mapFromArraysV1` 负责构造，回答的是“怎样用一组 keys 和一组 values 生成 map”；`valueFromMapV1` 负责按 key 取值，回答的是“运行时给出某个 key 后，怎样拿到对应 value”；`addOrUpdateMapEntryV1` 负责局部更新，回答的是“已有 map 上怎样补一个键值对，或覆盖某个 key 的 value”。对产品与调研读者来说，这说明 map 更像动态字典；对实现与开发读者来说，这也提示 map 场景的中心不是字段路径，而是键值关系。

### 典型输入与输出

这里同样只能提供说明性例子。假设输入里有一个 key 数组 `["country", "tier", "owner"]` 和一个 value 数组 `["CN", "gold", "ops"]`，业务希望先把它们构造成一个属性字典，再按 key 读取 `tier`，最后补一个新的 `status` 项。按正式清单结构，可推知一条典型写法会先用 `mapFromArraysV1(keys, values)` 生成 `Map<K, V>`，再用 `valueFromMapV1("tier", attr_map)` 读取 value，最后用 `addOrUpdateMapEntryV1("active", "status", attr_map)` 返回更新后的 map。

这个链路的输出也会在“整个 map”和“单个 value”之间来回切换。构造与更新阶段返回的是 `Map<K, V>`，按 key 读取阶段返回的是 `V`。这说明 map 的常见写法不是只把它当作最终对象保存起来，而是会在“整体容器”与“单个取值结果”之间反复切换，直到接到下游比较、展示或结构化拼接。

### 关键写法

`mapFromArraysV1` 的关键职责，是把两组并行数组压缩成一个键值映射。它的重要性不只是“能创建 map”，还在于它把 map 的来源写得很明确：key 和 value 不是散落定义的，而是由两组拉链式（zipped）数组共同决定。正式清单摘要里还明确提示 key 不能为 `null`，这说明构造阶段本身已经带有数据质量边界。

`valueFromMapV1` 则与 struct 的 `getStructFieldV2` 形成鲜明对比。后者依赖的是预先定义好的字段路径或 locator，前者依赖的是运行时提供的 key。也正因为如此，map 读取更适合处理“键集合不必事先固定成 schema 字段”的场景，而 struct 读取更适合处理“字段路径已经写死在对象结构里”的场景。

`addOrUpdateMapEntryV1` 负责局部更新，它让 map 能在保持整体容器不变的前提下补充或覆盖某个 key 的 value。这里还值得顺手保留一个实现边界：正式清单显示它的 `supported_in` 目前是 `Batch` 与 `Streaming`，不像前两个函数那样同时列出 `Faster`。本文不能据此推断全部运行时差异，但这至少提醒读者，map 局部更新能力在支持环境上可能比基础构造与读取更窄。

### 适用时机与边界

当问题天然是键值查找，例如属性字典、配置映射、动态标签集合或从 JSON 中抽出的不固定字段集合时，这个场景最合适。它尤其适合作为场景 9 的后续，因为 `parseJsonAsSchemaV3` 的目标 schema 之一就是 `Map<String, String>`，一旦 JSON 被解析成 map，后续往往就会进入本节的按 key 访问与局部更新。

边界上需要明确三点。第一，map 不是 struct 的动态同义词；它更适合 key 运行时才确定、或字段集合不希望被固定 schema 锁死的场景。第二，`valueFromMapV1` 解决的是按 key 查值，不会像 struct locator 那样表达多级固定路径。第三，`mapFromArraysV1` 的 key 不能为 `null`，而 `addOrUpdateMapEntryV1` 的支持环境当前也更窄，因此 map 构造与更新都不应被想象成“无条件适用的字典操作”。

### 当前证据

正式清单结构层面，`mapFromArraysV1`、`valueFromMapV1` 与 `addOrUpdateMapEntryV1` 都能在 `expression_inventory.json` 中稳定定位。`mapFromArraysV1` 同时归于 `Array` 与 `Map` 类别，参数是 key/value 两组数组，返回 `Map<K, V>`；`valueFromMapV1` 接收 `Key` 与 `Expression<Map<K, V>>`，返回 `V`；`addOrUpdateMapEntryV1` 接收新值、key 和原 map，返回更新后的 `Map<K, V>`，其 `supported_in` 当前列为 `Batch` 与 `Streaming`。

summary 结果层面，正式清单数量与唯一性已经闭合，但这三条记录的 `evidence_type` 仍统一为 `线索`。可稳定定位的页面字段层面，我们可以直接依赖类别组合、参数结构、返回类型、支持环境与官方链接。基于这些材料做出的判断是：map 场景应按“构造键值容器 -> 按 key 读取 value -> 局部更新 key-value”来理解，并且要与 struct 的固定字段路径访问明确区分开。

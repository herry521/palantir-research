# Data Integration Executive War Room 汇报页设计

## 摘要洞察

1. [决策] 汇报页采用“战情图型”而不是“模块清单型”：先给管理判断，再展示 Palantir 能力链路和状态点亮。
2. [决策] 主链路使用 Palantir 官方对象命名，面向熟悉 Palantir 的领导，减少概念翻译成本。
3. [决策] 每个节点展示“官方对象 + 中文解读 + 二级能力”，三级能力保留为展开态或附录，不进入主视觉。
4. [决策] 首屏关键数字使用“核心能力成熟度 / 对象域点亮 / 关键能力缺口 / 需关注风险”，直接支撑进展判断。
5. [决策] 页面默认服务领导汇报，配置能力保留但弱化，演示视图优先于编辑视图。

## 背景

现有 `Data Integration 能力地图` 已经具备 Palantir 对齐的能力数据和状态配置能力，但页面形态更接近“能力清单 + 配置工具”。领导汇报需要更强的叙事焦点：一眼看到当前进展、主链路完备性、风险集中区和下一步动作。

本设计将能力地图升级为 `Data Integration Executive War Room`，保留当前交互式状态配置能力，同时提供更适合汇报的首屏表达。

## 目标

- 将页面从“模块平铺”改为“Palantir 官方对象主链路战情图”。
- 支持已完成、进行中、风险、未开始的能力点亮。
- 让领导在首屏理解三个问题：现在到哪了、卡在哪里、下一步做什么。
- 保持状态仍可由用户配置，不把进展数据写死。
- 保留到仓库内 Palantir 术语和能力地图文档的来源引用。

## 非目标

- 不新增后端服务。
- 不引入复杂图表库。
- 不在主图展示所有三级能力。
- 不把业务价值指标作为默认主指标，除非后续补充真实业务数据。

## 页面结构

### 1. 顶部结论区

顶部使用正式汇报口径，采用“稳健达成型”语气：

> Data Integration 主链路已形成端到端闭环，下一阶段聚焦实时链路与对象服务补强。

该文案是默认示例，应允许后续在页面数据中配置或集中维护。

### 2. 关键数字区

首屏展示四个指标：

- 核心能力成熟度：核心能力中已完成项占比。
- 对象域点亮：主链路对象域中已达到可汇报状态的节点数量。
- 关键能力缺口：核心能力中未完成或进行中的缺口数量。
- 需关注风险：处于风险状态的能力数量或对象域数量。

数字需要从现有能力状态中计算，避免手工维护两套口径。

### 3. Palantir 主链路区

主链路使用 Palantir 官方对象作为节点标题：

1. Connectivity
2. Datasets
3. Pipeline
4. Builds
5. Lineage & Schedules
6. Ontology

每个节点包含：

- 官方对象名。
- 中文解读。
- 2-4 个二级能力摘要。
- 节点完成度。
- 状态视觉：完成、进行中、风险、未开始。

建议的节点解释：

| 节点 | 中文解读 | 二级能力摘要 |
| --- | --- | --- |
| Connectivity | 数据连接与同步 | Agents, Sources, Batch sync, Streaming sync, Exports |
| Datasets | 数据集与事务版本 | Datasets, Transactions, Streams, Branches |
| Pipeline | 数据加工开发 | Pipeline Builder, Code Repositories, Batch, Incremental, Streaming |
| Builds | 构建执行与新鲜度 | Jobs, JobSpec, Build resolution, Staleness |
| Lineage & Schedules | 血缘观测与调度 | Lineage graph, Build timeline, Triggers, Schedules |
| Ontology | 对象服务 | Object types, Object sets, Actions, Object Data Funnel |

### 4. 底部洞察区

主图下方使用三段式洞察：

- 关键进展：当前已打通或已成熟的主链路能力。
- 主要缺口：风险和未完成能力集中区。
- 下一步动作：下一阶段补齐方向和需要推进的管理动作。

洞察区默认使用示例文案，后续应支持集中配置，避免在 HTML 模板中分散维护。

### 5. 配置入口

导入 JSON、导出 JSON、重置等功能保留，但降低视觉权重：

- 默认汇报视图下只保留轻量配置入口。
- 演示模式隐藏编辑控件。
- 配置状态仍保存在 localStorage，并可导入/导出 JSON。

## 状态视觉

采用“色条 + 柔和底色 + 进度条”的点亮方式：

- 已完成：绿色顶部色条、浅绿色底色、完整或高比例进度条。
- 进行中：蓝色顶部色条、浅蓝底色、中等进度条。
- 风险：橙色顶部色条、浅橙底色、风险强调。
- 未开始：灰色顶部色条、低饱和灰底、弱化显示。

不采用高对比发光风格，避免页面过于驾驶舱化；不采用纯徽标风格，避免点亮效果不足。

## 数据与计算

现有能力数据可继续作为基础，但需要增加一个“汇报节点”映射层：

- 每个 Palantir 主链路节点映射到若干现有模块或能力项。
- 节点进度由映射能力项状态聚合计算。
- 全局关键数字由主链路节点和核心能力状态聚合计算。

建议新增字段或结构：

```js
const DI_REPORT_NODES = [
  {
    id: "connectivity",
    name: "Connectivity",
    zhName: "数据连接与同步",
    capabilityRefs: [
      "data-connection.agents",
      "data-connection.sources",
      "data-connection.batch-sync",
      "data-connection.streaming-sync",
      "data-connection.exports"
    ]
  }
];
```

## 响应式策略

- 桌面宽屏：顶部结论和关键数字左右布局，主链路六列展示。
- 中等屏幕：主链路三列或横向滚动，保证节点不挤压。
- 移动端：结论、指标、节点、洞察纵向堆叠。
- 演示模式：优先保持一屏表达，隐藏导航和配置按钮。

## 验证标准

- 首屏不需要滚动即可看到结论、关键数字和主链路上半部分。
- 演示模式下页面不像配置工具。
- 状态导入后，主链路节点和关键数字同步更新。
- 风险状态在主图和洞察区都能被识别。
- 术语与仓库内 Palantir 对齐文档保持一致。

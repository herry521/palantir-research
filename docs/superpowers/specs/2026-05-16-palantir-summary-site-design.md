# Palantir 调研总结 HTML 专题页设计文档

> 创建日期：2026-05-16
> 目标：基于现有 Palantir Foundry / Pipeline 调研材料，产出一套可离线打开的静态 HTML 专题页
> 主要受众：管理层（首页一页纸） + 研发团队（详情页）
> 设计定位：管理结论先行，研发证据下钻，整体视觉风格参照“理想汽车新能源深绿”

---

## 1. 设计目标

本次交付不是把已有 Markdown 调研文档搬进网页，而是将调研成果重组为一套适合汇报和传播的静态专题页：

- 对管理层：在 5 分钟内读懂这轮调研的核心结论、关键差距和建设建议
- 对研发团队：沿着首页结论下钻到具体机制、能力边界和设计启发
- 对后续维护：页面结构和视觉系统可复用，后续新增专题时不需要推翻现有框架

本次设计遵循以下原则：

1. **结论先于细节**：首页优先回答“说明了什么”“该做什么”，而不是铺满技术事实。
2. **一页纸保持克制**：首页承担判断和导航，不承担完整技术解释。
3. **研发页强调可扫读**：详情页以结论块、机制块和来源块组织，而非长篇连续叙述。
4. **离线可交付**：所有页面以本地静态 HTML/CSS/少量原生 JS 方式实现，不依赖运行时服务。
5. **视觉统一但层级分明**：首页偏深色、高层汇报感；详情页更注重阅读效率和技术信息组织。

---

## 2. 交付结构

交付形式采用“1 个管理首页 + 1 个研发总览页 + 5 个专题页”的静态站点结构。

建议目录结构如下：

```text
deliverables/
├── index.html
├── styles.css
├── app.js
├── assets/
│   ├── diagrams/
│   └── images/
└── pages/
    ├── overview.html
    ├── expression-and-operators.html
    ├── execution-and-incremental.html
    ├── streaming-architecture.html
    ├── lineage-ontology-governance.html
    └── engineering-and-ecosystem.html
```

说明：

- `index.html`：管理层一页纸首页
- `pages/overview.html`：研发总览页，作为所有技术专题入口
- `pages/*.html`：专题详情页
- `styles.css`：共用视觉系统、布局、卡片、导航、图表容器样式
- `app.js`：少量交互脚本，如页内导航高亮、目录展开、卡片悬停增强
- `assets/`：复用现有导出的 PNG 架构图，必要时补充轻量信息图

## 3. 首页信息架构（管理层一页纸）

首页采用“结论先行型”结构，阅读路径从“战略判断”自然过渡到“能力差距”和“建设建议”。

### 3.1 区块结构

首页自上而下包含 5 个区块：

1. **一句话结论**
   直接回答本轮调研的总判断：Palantir 的优势不只是算子丰富，而是把表达层、执行层、治理层和应用层做成了闭环的平台能力。

2. **三个管理判断**
   使用 3 张高密度结论卡片分别说明：
   - Palantir 强项的本质是什么
   - 我们当前差距最集中的环节在哪里
   - 我们的建设策略应当优先补什么，而不是全面复制什么

3. **能力差距总览**
   用一张 4 维能力地图或 4 组对比卡展示差距：
   - 表达与开发体验
   - 执行与流批能力
   - 治理与语义层
   - 工程化与生态集成

4. **建设建议**
   采用三阶段表达：
   - 先补底座
   - 再做增强
   - 最后做平台闭环

5. **进入技术详情**
   在首页底部提供研发总览页和专题页入口，控制信息密度，让首页保持“一页纸”属性。

### 3.2 首页内容目标

首页不追求“讲全”，只追求管理层看完后形成三个清晰认知：

- 这次调研发现的不是单点功能，而是一套平台化方法论
- 我们的差距不是某个算子或某个引擎，而是多层能力尚未闭环
- 下一步建设需要有先后顺序，优先投入平台底座与关键链路

## 4. 技术详情页结构（研发团队）

研发侧采用“1 个总览页 + 5 个专题页”的混合方式。

### 4.1 总览页

`pages/overview.html` 承担研发地图页角色，目标是帮助读者快速建立整体理解并进入具体专题。

总览页包含以下内容：

- 平台能力分层图
- 5 个专题入口卡片
- 每个专题对应的核心问题
- 与首页结论的映射关系

总览页不重复首页文案，而是回答“如果要拆开看，这套能力分别包含什么”。

### 4.2 专题页划分

#### 1. `expression-and-operators.html`

主题：表达层、Transform 抽象、算子平台、开发体验

回答问题：

#### 2. `execution-and-incremental.html`

主题：Spark、Faster、增量批处理、构建调度

回答问题：
- 它为什么能在成本、性能和开发复杂度之间取得平衡
- 增量链路如何成为流处理与全量批处理之间的中间层

主要来源：

#### 3. `streaming-architecture.html`

#### 4. `lineage-ontology-governance.html`

### 4.3 专题页统一模板

为保证所有研发详情页读法一致，每个专题页统一采用 4 段结构：

1. **结论**
   用 2 到 4 条结论卡概括本专题最重要的判断。

2. **关键机制**
   解释技术原理、能力边界和平台实现逻辑，可配图或轻量流程图。

3. **对我们的启发**
   将调研结论转化为平台设计建议、建设顺序或取舍建议。

4. **相关调研来源**
   以可点击来源卡或列表形式指向仓库内文档，方便研发继续深挖。

## 5. 视觉与交互设计

整体视觉方向参照“理想汽车新能源深绿”，但不复制品牌页面，而是提炼出适合技术汇报的视觉语言。

### 5.1 气质定位

- 克制、专业、偏战略汇报感
- 不是营销风，不做夸张动画和视觉噱头
- 首页强调“判断力”，详情页强调“阅读效率”

### 5.2 色彩系统

建议使用以下色彩层级：

- **主背景**：深绿或灰绿系，用于首页首屏和重点结论区域
- **辅背景**：雾白、浅灰绿，用于详情页阅读区
- **主强调色**：高纯度绿色，用于关键数字、重点标签、章节锚点
- **次强调色**：低饱和金色或浅青绿色，用于差异点和提醒信息
- **文字色**：深灰 + 雾白双系统，适配深浅背景

### 5.3 页面风格

首页：

- 大留白标题区
- 强结论卡片
- 深色背景下的信息模块
- 少量具有“平台分层感”的几何线框或数据网格纹理

详情页：

- 更亮、更适合长时间阅读
- 左侧或顶部章节导航
- 统一的结论块、机制块、来源块组件
- 对图表和表格做轻量品牌化包裹，而不是原样贴图

### 5.4 交互策略

只保留轻交互：

- 页内锚点跳转
- 卡片悬停高亮
- 导航当前章节状态
- 首页与详情页之间的轻量跳转引导

不引入复杂动画、全屏滚动特效或外部依赖库。

## 6. 内容重组策略

HTML 页面需要从“文档仓”变成“汇报型内容产品”，因此必须重组内容，而不是直接转贴。

重组原则如下：

1. **首页所有文案都重新写**
   不直接复制综合文档段落，保证表达更短、更适合管理层。

2. **详情页做主题压缩**
   研发页可以引用原始观点，但要按“结论-机制-启发-来源”重组，避免大段流水账。

3. **图优先于长文**
   现有 `diagrams/` 下导出的 PNG 优先用于支撑机制说明，必要时追加简单信息图和能力地图。

4. **突出“对我们的启发”**
   每个专题页都要显式回答“这对我们意味着什么”，避免变成纯外部调研摘录。

## 7. 实现边界

本次实现范围包括：

- 一套可直接打开的静态 HTML 页面组
- 共用样式与轻量脚本
- 首页 + 总览页 + 5 个专题页
- 基于现有调研材料的内容重组与视觉呈现

本次不包含：

- 将全部 Markdown 自动转换为网页知识库
- 完整全文检索系统
- 后端服务、构建服务或 CMS
- 复杂图表库驱动的交互可视化
- 多语言切换

---

## 8. 验收标准

交付完成后，应满足以下标准：

1. 管理层首页在单页内完成核心表达，且首屏能读懂总判断。
2. 研发团队可以通过总览页进入 5 个专题页，并快速定位对应主题。
3. 所有页面可离线打开，无需额外依赖和本地服务。
4. 视觉风格统一，首页和详情页既同源又有层次区分。
5. 所有专题页都清晰包含“结论、关键机制、对我们的启发、相关来源”四段。
6. 页面内容与仓库现有调研文档保持一致，不引入未证实的新结论。

---

## 9. 主要风险与控制

1. **信息过载风险**
   若首页承载过多技术细节，会损失“一页纸”价值。控制方式是将细节严格下沉到详情页。

2. **页面像文档搬运风险**
   若详情页只是长文复制，会失去 HTML 交付的意义。控制方式是统一组件化重组。

3. **视觉过度品牌化风险**
   若过度靠近汽车营销页，会削弱技术汇报的可信度。控制方式是保留深绿气质，但用克制的布局和动效。

4. **专题边界重叠风险**
   增量、流式、治理等主题天然有交叉。控制方式是每页围绕“核心问题”组织，而不是围绕所有事实。

---

## 10. 下一步实现建议

在本设计文档确认后，进入实现阶段时建议按以下顺序推进：

1. 先搭建静态站点骨架与共用视觉系统
2. 完成管理层首页
3. 完成研发总览页
4. 依次补齐 5 个专题页
5. 最后统一做链接、样式、内容一致性检查

该顺序可以保证最先交付最有价值的页面，并降低“全量铺开后再返工”的风险。
